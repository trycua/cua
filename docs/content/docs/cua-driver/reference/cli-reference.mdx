Cross-platform computer-use automation driver — MCP stdio server plus a shell CLI. The same binary serves macOS (Swift driver) and Windows / Linux (`cua-driver-rs`).

`cua-driver --help` prints the authoritative subcommand list; `cua-driver --version` prints the version. The subcommands are: `mcp`, `list-tools`, `describe`, `call`, `serve`, `stop`, `status`, `mcp-config`, `config`, `recording`, `update`, `check-update`, `doctor`, `diagnose`, `autostart`, `skills`.

<Callout type="warn">
**Passing JSON on Windows PowerShell 5.1.** PowerShell 5.1 strips the quotes around JSON field names in multi-field arguments. Pipe the JSON via stdin instead, or use PowerShell 7+ (`pwsh`):

```powershell
'{"pid":1234,"window_id":5678}' | cua-driver call get_window_state
```
</Callout>

## Tool dispatch

### cua-driver call

Invoke an MCP tool directly from the shell — the same code path the MCP server uses.

```bash
cua-driver call <tool-name> '<json-args>'
```

`<json-args>` is a JSON object matching the tool's input schema. When it's omitted and stdin is a pipe, the JSON is read from stdin; when neither is provided, the tool is called with no arguments. When a daemon is running, `call` forwards to it (so the per-pid `element_index` cache is shared across invocations); use `--socket` to target a non-default daemon socket / pipe.

```bash
cua-driver call list_apps
cua-driver call launch_app '{"name":"Calculator"}'
echo '{"pid":844,"window_id":10725}' | cua-driver call get_window_state
```

### cua-driver list-tools

List every MCP tool with its one-line description.

### cua-driver describe

Print a tool's full description and JSON input schema.

```bash
cua-driver describe click
```

## Daemon management

### cua-driver mcp

Run the stdio MCP server. When spawned from an IDE terminal, `mcp` may proxy tool calls to a running `cua-driver serve` daemon so they execute under the right process identity — on macOS to inherit `CuaDriver.app`'s TCC grants, on Windows to reach the interactive desktop from a Session 0 SSH login. Tool semantics are identical to the in-process path. See the [process model](/cua-driver/guide/getting-started/process-model) and [SSH on Windows](/cua-driver/guide/getting-started/windows-ssh).

**Flags / options:**

| Name | Description |
| ---- | ----------- |
| `--no-daemon-relaunch` | Stay in-process; skip the daemon proxy. Also settable via `CUA_DRIVER_RS_MCP_NO_RELAUNCH=1`. |
| `--socket <path>` | Override the daemon socket / pipe path used by the proxy. |
| `--experimental-pip` | Show an always-on-top window with the latest post-action screenshot + a one-line label. macOS only today; Windows / Linux print a not-yet-implemented notice. See [PiP Preview](/cua-driver/guide/getting-started/pip-preview). |
| `--experimental-pip-geometry <WxH[+X+Y]>` | Override the PiP window size / position. Default `480x360` in the top-right corner. |

### cua-driver serve

Run cua-driver as a long-running daemon. Subsequent `cua-driver call` / `mcp` invocations auto-detect the daemon and forward to it, so the per-pid `element_index` cache survives across CLI calls. Accepts the same `--experimental-pip` / `--experimental-pip-geometry` flags as `mcp`.

### cua-driver stop

Ask the running daemon to exit gracefully.

### cua-driver status

Report whether a daemon is running, and its socket / pipe path and pid.

### cua-driver mcp-config

Print the MCP server config JSON, or a client-specific snippet with `--client <name>` (`claude`, `codex`, `cursor`, `antigravity`, `openclaw`, `opencode`, `hermes`, `pi`; `gemini` is a legacy alias for `antigravity`). See [Integrations](/cua-driver/guide/getting-started/integrations).

```bash
cua-driver mcp-config --client cursor
```

## Trajectory recording

### cua-driver recording

Control the trajectory recorder on a running daemon (recording state lives in the daemon, so these require `cua-driver serve`). Wraps the `start_recording` / `stop_recording` / `get_recording_state` tools with human-friendly output.

```bash
cua-driver recording start ~/cua-trajectories/demo1   # video on by default
cua-driver recording status
cua-driver recording stop
```

Video capture is on by default (macOS uses ScreenCaptureKit; Windows / Linux use an ffmpeg subprocess). See the [`start_recording` tool](/cua-driver/reference/mcp-tools#start_recording) for the per-turn folder layout and the `record_video` opt-out.

## Configuration

### cua-driver config

Read and write the persistent driver config. Values take effect immediately and survive daemon restarts.

```bash
cua-driver config get capture_mode
cua-driver config set capture_mode som
cua-driver config set update_check_enabled false
```

Known keys: `capture_mode` (`vision` | `ax` | `som`), `max_image_dimension` (integer), `update_check_enabled` (boolean), `experimental_pip` (boolean), `experimental_pip_geometry` (string). Telemetry can also be toggled per-environment with `CUA_DRIVER_RS_TELEMETRY_ENABLED=0`.

## Updating

### cua-driver check-update

Ask GitHub whether a newer release is available. Read-only — never installs. Human-friendly text by default.

| Name | Description |
| ---- | ----------- |
| `--json` | Emit a machine-readable payload (same shape as the `check_for_update` MCP tool). |
| `--no-cache` | Skip the 20-hour on-disk cache and force a fresh GitHub round-trip. |

### cua-driver update

The same check as `check-update`, then suggests `--apply` when outdated.

| Name | Description |
| ---- | ----------- |
| `--apply` | Download and install the latest release via the canonical installer. |
| `--json` | Emit the structured check payload (does not change `--apply` behavior). |

See [Updating](/cua-driver/guide/getting-started/updating) for the full workflow.

## Diagnostics

### cua-driver doctor

Run a structured environment + install-health report — version, install layout, telemetry, and platform-specific probes (TCC + LaunchAgent on macOS; UI Automation + interactive desktop session on Windows; X11/Wayland + AT-SPI on Linux). Pass `--json` for a scripting-friendly shape; the exit code is non-zero only when a probe reports an error.

### cua-driver diagnose

Print a paste-able support report (bundle path / cdhash / TCC status on macOS; equivalent identity + permission details elsewhere) for attaching to issues.

## Autostart

### cua-driver autostart

Keep a `cua-driver serve` daemon running in the user's interactive session across logons (Windows-only today; macOS / Linux point at the manual recipe). See [Autostart](/cua-driver/guide/getting-started/autostart).

| Subcommand | Description |
| ---- | ----------- |
| `enable` | Register a logon Scheduled Task so `serve` starts at every interactive logon. |
| `disable` | Remove the autostart entry. No-op if not registered. |
| `status` | Print whether the entry is registered and whether the daemon is running. |
| `kick` | Start the entry now without re-logging. |

## Skills

### cua-driver skills

Manage the agent skill pack — symlink the versioned SKILL.md into each detected agent's skills directory (Claude Code, Codex, OpenClaw, OpenCode).

| Subcommand | Description |
| ---- | ----------- |
| `install` | Fetch the skill pack from GitHub Releases and symlink it into each agent. Idempotent; never overwrites existing user links. Add `--from main` to fetch from the main branch instead of the tagged release. |
| `update` | Re-fetch the skill pack, refreshing the local copy and links. |
| `uninstall` | Remove the agent symlinks. Add `--all` to also delete the local copy. |
| `status` | Report local install state and per-agent link state. Read-only. |
| `path` | Print where the local skill pack lives. |

## Global options

Available on all commands:

- `--help` — Show help information (the authoritative subcommand + flag list).
- `--version` — Show the version number.
