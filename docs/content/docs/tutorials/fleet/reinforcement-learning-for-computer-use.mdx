# Reinforcement learning for computer use

This tutorial explains the repository-level off-policy GRPO-style training loop in
`cua_bench.trainer.off_policy.tinker.rl_loop`. It collects or reuses Cua Bench trajectories, derives
terminal rewards from task evaluators, and trains a LoRA through Tinker.

<Callout type="warn">
  This is experimental repository code, not a stable public Fleet or Cua Bench CLI. Its online mode
  shells out to `cb run dataset`; its Fleet use depends on a session/worker adapter that this
  checkout does not provide.
</Callout>

## Learning outcome

You can define separate train and held-out computer-use tasks, collect trace-compatible rollouts,
run the current GRPO/Tinker module conservatively, and evaluate checkpoints without leaking rewards
or leaving Fleet claims behind.

## Prerequisites

- A source checkout containing `libs/cua-bench`, its optional Tinker dependencies, and a valid
  `TINKER_API_KEY`.
- A base model, a reachable `vLLM`-compatible endpoint, and budget limits for training and rollouts.
- Fleet access and a fixed pool image digest/resources when rollouts will use Fleet. See
  [Fleet images](/concepts/fleet/images) and [Use the Fleet SDK workflow](/how-to-guides/fleet/use-the-sdk).
- A Cua Bench Fleet-backed `DesktopSession`/worker adapter. Cua Bench currently exposes `native` and
  `simulated` desktop sessions; its CUA Cloud provider is not a Fleet provider.

## 1. Define train and held-out tasks

Create `tasks/three_lines/main.py`. The train and `test` splits intentionally use different expected
documents. The evaluator is identical for both and reads only the document the agent saved. Do not
put a reference answer, evaluator source, reward endpoint, or a completion marker in the guest image.

```python
import base64
import shlex

import cua_bench as cb


def make_task(task_id: str, lines: list[str]) -> cb.Task:
    expected_text = "\n".join(lines) + "\n"
    description = (
        "Using the visible desktop text editor, create ~/Desktop/answer.txt with exactly these lines "
        f"in order: {', '.join(lines)}. Save the document and leave it on the Desktop."
    )
    return cb.Task(
        task_id=task_id,
        description=description,
        metadata={"expected_text": expected_text},
        computer={
            "provider": "native",
            "setup_config": {"os_type": "linux", "width": 1024, "height": 768},
        },
    )


@cb.tasks_config(split="train")
def load_train_tasks():
    return [
        make_task("three-lines-train-1", ["amber", "cobalt", "dune"]),
        make_task("three-lines-train-2", ["fern", "garnet", "harbor"]),
    ]


@cb.tasks_config(split="test")
def load_test_tasks():
    return [
        make_task("three-lines-test-1", ["indigo", "topaz", "willow"]),
        make_task("three-lines-test-2", ["violet", "xenon", "zephyr"]),
    ]


async def reset_desktop(session: cb.DesktopSession):
    await session.run_command("mkdir -p ~/Desktop && rm -f ~/Desktop/answer.txt")


@cb.setup_task(split="train")
async def set_up_train(task_cfg: cb.Task, session: cb.DesktopSession):
    await reset_desktop(session)


@cb.setup_task(split="test")
async def set_up_test(task_cfg: cb.Task, session: cb.DesktopSession):
    await reset_desktop(session)


async def score(task_cfg: cb.Task, session: cb.DesktopSession) -> list[float]:
    expected = base64.b64encode(task_cfg.metadata["expected_text"].encode()).decode("ascii")
    script = (
        "import base64, pathlib, sys; "
        "expected = base64.b64decode(sys.argv[1]); "
        "path = pathlib.Path.home() / 'Desktop' / 'answer.txt'; "
        "actual = path.read_bytes() if path.exists() else b''; "
        "raise SystemExit(0 if actual == expected else 1)"
    )
    command = f"python3 -c {shlex.quote(script)} {shlex.quote(expected)}"
    result = await session.run_command(command, check=False)
    return [1.0 if result["return_code"] == 0 else 0.0]


@cb.evaluate_task(split="train")
async def evaluate_train(task_cfg: cb.Task, session: cb.DesktopSession) -> list[float]:
    return await score(task_cfg, session)


@cb.evaluate_task(split="test")
async def evaluate_test(task_cfg: cb.Task, session: cb.DesktopSession) -> list[float]:
    return await score(task_cfg, session)
```

`native` in this module is a source-backed local provider name, not a Fleet configuration. It makes
the task locally checkable. For Fleet rollouts, the adapter must create an equivalent fresh desktop
for each reset while preserving the task description, setup, evaluator, resolution, image digest,
and resource contract.

Keep the test split unavailable to prompt editing, adapter development, and training-rollout
selection. The loop trains from terminal evaluator rewards, so any secret expected value accessible
to the agent is reward leakage rather than learning.

## 2. Understand the Fleet adapter boundary

Online rollouts cannot run on Fleet merely by setting a provider string or using `--provider cloud`:
neither is a Fleet integration in this checkout. A Fleet adapter must:

1. Create one isolated claim per rollout from the pinned pool image and resource configuration.
2. Adapt the claimed desktop to Cua Bench's `DesktopSession` protocol, including screenshots, action
   execution, guest commands used by setup/evaluation, and `close()`.
3. Ensure every reset starts from a fresh claim and serialize ordinary Cua Bench trace events:
   `reset`, `agent_step`, and `evaluate`, including available screenshots.
4. Close the session and delete the claim on success, timeout, cancellation, and exception.

Once implemented, Fleet supplies the parallel isolated claims; Cua Bench's rollout loop supplies the
repeated `cb run dataset` calls. Cap parallelism at the number of claims and model requests your
Fleet pool and providers can safely support.

## 3. Prepare trace-compatible rollouts

Each episode loaded by `traces.py` comes from a Cua Bench run output directory. It requires a task
trace with an `evaluate` event, reads that event's scalar or first-list-item result as the terminal
reward, obtains the task text and initial screenshot from `reset`, and uses `agent_step` records for
actions and later screenshots.

The practical consequences are:

- A failed or incomplete run without `evaluate` is skipped rather than converted into a trainable
  episode.
- Keep terminal rewards stable, bounded, and computed after the task ends. This simple task uses
  `0.0` or `1.0`; record evaluator versions with the run.
- Use at least two rollouts per task. Group-relative GRPO advantage is each terminal reward minus the
  mean reward for episodes sharing the task description, so a group size below two is not meaningful.
- `--max-images` keeps only the most recent screenshots with each trajectory. Start at `3`; increasing
  it can improve context but raises image/token and training cost.
- Fix image digests, resolution, task ordering, text, evaluator code, max steps, timeouts, and model
  sampling settings where available. Non-determinism from a desktop, coding agent, or service should
  be reported, not silently absorbed into rewards.

Do not hand arbitrary Fleet screenshots or logs to offline mode. They must be Cua Bench run artifacts
in the trace format loaded by `traces.py`.

## 4. Gate the experimental online helper

Do not use `rl_loop --mode online` from this checkout. Its helper shells out to `cb run dataset` using
`--with libs/python/agent/agent`, but this checkout contains the package at `libs/python/agent`; the
hard-coded `libs/python/agent/agent` path does not exist. Online mode is therefore not runnable here,
even after you implement a Fleet-backed `DesktopSession`/session provider and registered agent, until
upstream corrects `rollout.py`.

The helper is also unsafe to treat as a conservative default: it polls for up to 3600 seconds and
raises `TimeoutError` when that deadline expires, but it does not issue `cb run stop` or otherwise
cancel the run. That can leave sessions and Fleet claims alive and billing. It also creates the Cua
Bench environment with the default `train` split; neither `cb run dataset` nor `rl_loop` exposes a
split-selection flag. Configure model endpoints through the selected adapter environment; although
`rl_loop` parses and stores `--vllm-url`, it does not wire that value into rollout collection or
training.

## 5. Collect safe rollout runs, then train offline

The current practical route is offline-only: collect valid Cua Bench runs directly with `cb`, inspect
and clean them, then pass their run IDs to `rl_loop --mode offline`. This still requires an implemented
registered Fleet adapter and Fleet-backed session/provider in the direct `cb run dataset` path; the
command does not configure Fleet pools, image digests, resources, or claim timeouts itself.

Collect one small train dataset at a time with one parallel task runner and a low action limit. Omit
`--output-dir`: offline mode resolves run IDs only beneath `$XDG_DATA_HOME/cua-bench/runs/<run-id>`
(or `~/.local/share/cua-bench/runs/<run-id>`), so a custom output directory would not be discovered by
the current offline loader.

```bash
export FLEET_AGENT_NAME='your-registered-fleet-agent'
export FLEET_MODEL='your-adapter-model-id'

cb run dataset ./tasks/three_lines \
  --agent "$FLEET_AGENT_NAME" \
  --model "$FLEET_MODEL" \
  --max-steps 20 \
  --max-parallel 1
```

Record the emitted run ID. Inspect each run before collecting the next rollout:

```bash
cb run info <run-id>
cb run watch <run-id>
cb run logs <run-id>
# If it is stuck, cancel it and then delete any Fleet claims it left behind.
cb run stop <run-id>
```

Wait for completion, verify that every required trajectory has an `evaluate` event and the expected
trace directory, and check for leaked Fleet claims. Delete any leaked claims before starting another
collection run or training. Collect at least two distinct completed run IDs whose trajectories cover
each train task: GRPO needs at least two trajectories for each matching task description.

Then run offline training. `--num-rollouts 2` remains a parser/configuration value in offline mode but
does not collect runs; the distinct `--offline-run-ids` and their trace trajectories provide the
required group size instead.

```bash
export TINKER_API_KEY='replace-with-your-key'

uv run --project libs/cua-bench python -m cua_bench.trainer.off_policy.tinker.rl_loop \
  --mode offline \
  --base-model Qwen/Qwen3-8B \
  --tasks-path ./tasks/three_lines \
  --agent unused-in-offline-mode \
  --model unused-in-offline-mode \
  --max-steps 20 \
  --epochs 1 \
  --save-every 1 \
  --log-path ./artifacts/tinker-three-lines-offline \
  --num-rollouts 2 \
  --offline-run-ids <first-completed-run-id> <second-completed-run-id> \
  --gamma 1.0 \
  --lr 1e-5 \
  --beta1 0.9 \
  --beta2 0.95 \
  --max-images 3
```

Offline mode reuses run IDs every epoch and loads only compatible Cua Bench trace directories. It does
not accept arbitrary Fleet screenshots or logs, and it does not invoke the supplied agent/model strings
or a model endpoint.

To resume this checkout's loop, rerun it with the same `--log-path`. `run()` finds the latest saved
checkpoint there and starts from its saved epoch. `--resume-from` is parsed into the configuration but
currently unused, so do not pass it as a resume mechanism.

## 6. Evaluate the held-out split and checkpoints

Neither `cb run` nor `rl_loop` can select `split="test"`. Use a source-level evaluation loop instead.
The following uses verified `cua_bench.make(task_path, split="test")`, the registered-agent lookup,
`Environment.reset()`, `BaseAgent.perform_task(...)`, `Environment.evaluate()`, and `Environment.close()`.
It runs after your Fleet `DesktopSession`/provider adapter is wired into the task's `native` path.

`rl_loop` checkpoints are Tinker training-state handles. They are not automatically served inference
endpoints and the Fleet adapter does not automatically consume them. Before evaluating, export or
deploy the baseline and trained state through your own Tinker/model-serving workflow, then configure
the registered Fleet agent adapter to target the corresponding served model identifier or endpoint.
This repository supplies no command that deploys a checkpoint for inference.

Save this as `held_out_eval.py`. `SERVED_MODEL` is the identifier the adapter actually targets, while
`CHECKPOINT_STATE_PATH` is report metadata such as the Tinker state path used to create that deployment.

```python
import asyncio
import os
from pathlib import Path

from cua_bench import make
from cua_bench.agents import get_agent
from cua_bench.config.loader import ConfigLoader


TASK_PATH = Path("./tasks/three_lines")
AGENT_NAME = os.environ["FLEET_AGENT_NAME"]
SERVED_MODEL = os.environ["SERVED_MODEL"]
CHECKPOINT_STATE_PATH = os.environ["CHECKPOINT_STATE_PATH"]
TASK_ID = int(os.environ["TASK_ID"])
MAX_STEPS = 20
OUTPUT_DIR = Path(os.environ["OUTPUT_DIR"])


async def main() -> None:
    agent_class = get_agent(AGENT_NAME, ConfigLoader())
    if agent_class is None:
        raise RuntimeError(f"Unknown registered agent: {AGENT_NAME}")

    env = make(str(TASK_PATH), split="test")
    env.max_steps = MAX_STEPS
    OUTPUT_DIR.mkdir(parents=True, exist_ok=True)
    try:
        _, task_cfg = await env.reset(task_id=TASK_ID)
        agent = agent_class(model=SERVED_MODEL)
        agent_result = await agent.perform_task(
            task_description=task_cfg.description,
            session=env.session,
            logging_dir=OUTPUT_DIR,
        )
        reward = await env.evaluate()
        print(
            {
                "task_id": TASK_ID,
                "checkpoint_state_path": CHECKPOINT_STATE_PATH,
                "served_model": SERVED_MODEL,
                "reward": reward,
                "failure_mode": agent_result.failure_mode.value,
            }
        )
    finally:
        await env.close()


asyncio.run(main())
```

Set two distinct served-model identifiers. The shell guard prevents an accidental same-model
comparison; replace the example state paths with the recorded baseline state and trained checkpoint
state that your serving workflow deployed. This tutorial's `load_test_tasks()` defines exactly two
held-out variants, so the source-level task indices are `0` and `1`.

```bash
export FLEET_AGENT_NAME='your-registered-fleet-agent'
export BASELINE_MODEL='your-served-baseline-model'
export TRAINED_MODEL='your-served-trained-model'

if [ "$BASELINE_MODEL" = "$TRAINED_MODEL" ]; then
  echo 'BASELINE_MODEL and TRAINED_MODEL must differ' >&2
  exit 1
fi

for TASK_ID in 0 1; do
  TASK_ID="$TASK_ID" \
  SERVED_MODEL="$BASELINE_MODEL" \
  CHECKPOINT_STATE_PATH='baseline-state-or-release-id' \
  OUTPUT_DIR="./artifacts/held-out-eval/baseline/task-${TASK_ID}" \
  python3 held_out_eval.py

  TASK_ID="$TASK_ID" \
  SERVED_MODEL="$TRAINED_MODEL" \
  CHECKPOINT_STATE_PATH='tinker://trained-state-path' \
  OUTPUT_DIR="./artifacts/held-out-eval/trained/task-${TASK_ID}" \
  python3 held_out_eval.py
done
```

The loop uses a unique `OUTPUT_DIR` for every model/task pair. Run it under the same frozen Fleet
contract and evaluator. Never use held-out rewards to select rollout trajectories. Record the task ID,
checkpoint/state path, served model identifier, test success rate, mean reward, latency, failure modes,
and final screenshots with the results; otherwise it is possible to silently compare the same model
twice or omit a held-out task.

The loop saves periodic state checkpoints before later epochs and a final checkpoint under
`--log-path`. Verify a checkpoint exists before claiming that training is resumable. Bound the first
experiment to one epoch, two rollouts, a low max-step count, a fixed image limit, and a small task
set; then review token, image, Fleet-capacity, and Tinker charges before scaling out.

## 7. Clean up claims and artifacts

The Fleet adapter must release every claim in `finally`, including claims from failed model calls and
interrupted training. After stopping the loop, list claims created by its run label and delete any
stragglers using the lifecycle in [Use the Fleet SDK workflow](/how-to-guides/fleet/use-the-sdk).
Stop unfinished Cua Bench runs with `cb run stop <run-id>`.

Retain trace-compatible run directories, evaluator versions, the frozen Fleet contract, checkpoint
metadata, and held-out reports needed for reproducibility. Delete temporary credentials and any
unneeded screenshots, desktop logs, or copied task data according to your retention policy.

## What happened

The experimental loop either collected Cua Bench runs online or loaded compatible runs offline. It
converted evaluator terminal rewards and recent trajectory screenshots into group-relative GRPO
training data, while Fleet remained an explicit desktop/claim adapter boundary rather than an
implicit Cua Bench provider.
