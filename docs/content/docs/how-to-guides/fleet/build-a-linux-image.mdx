This guide defines the intended portable workflow for a Linux sandbox image:

1. Describe and build the image on a local machine with the Sandbox SDK.
2. Publish the completed image as a flattened artifact in an OCI-compatible registry.
3. Load that artifact on another local machine with `Image.from_registry()`.

<Callout type="warn" title="Status: target workflow, not an end-to-end supported workflow">
  The Sandbox SDK does not yet provide a supported public end-to-end workflow that builds a Linux
  SDK image, publishes the resulting portable artifact to a registry, and boots it locally from that
  registry reference. `Image.from_registry()` exists, but the matching public publishing and
  flattening interface is not available yet. Treat the publishing steps and expected results on this
  page as the contract for future implementation, not as commands you can run today.

`Sandbox.snapshot()` is unavailable in cloud environments. Build and package the image locally
instead of using a cloud sandbox as the source artifact.

</Callout>

## Prerequisites

For the complete workflow, you will need:

- A local machine capable of running Linux sandboxes.
- The Cua Sandbox SDK and its local virtualization dependencies.
- An OCI-compatible registry that can store the portable artifact.
- Credentials with permission to push to and pull from the target repository.

The SDK can already describe Linux images and load registry references. The public SDK operation
that publishes a completed local image is still pending.

## 1. Describe the image locally

Use the public `Image` builder to declare the Linux base, packages, files, environment variables, and
setup commands that the sandbox needs. Keep the definition reproducible: pin important package or
artifact versions, avoid embedding secrets, and make setup commands fail when installation fails.

For example, the image definition may use existing builder methods such as `Image.linux()`,
`.apt_install()`, `.copy()`, `.env()`, and `.run()`. See [Build custom images](/how-to-guides/sandbox/images)
for the currently available builder API.

**Expected behavior:** the SDK turns the image definition into a bootable local sandbox environment.
Building the same pinned definition should produce an environment with the same installed tools and
configuration.

## 2. Verify the local sandbox

Boot the image locally and test the behavior that downstream users will depend on. Verify at least:

- The sandbox reaches a ready state and accepts SDK operations.
- Required applications, packages, files, and environment variables are present.
- Setup services start after a clean reboot when the workflow depends on them.
- No build credentials, temporary files, or sensitive shell history remain in the image.

Fix the image definition and rebuild until these checks pass. The artifact published later should
represent this tested state, not an unverified intermediate build.

**Expected behavior:** a fresh local sandbox created from the completed image passes the same smoke
tests without additional manual setup.

## 3. Produce a portable flattened artifact

The intended publishing flow converts the tested local image into one self-contained artifact. The
artifact must include the complete bootable disk state rather than depend on local backing files,
build caches, or copy-on-write layers that exist only on the build machine.

The public SDK does not yet expose the command or method that performs this flattening step. Until
that interface exists, this guide intentionally does not prescribe a command name, method signature,
file layout, or media type.

**Expected behavior:** the flattened artifact can be copied to another machine and booted without
access to the original build machine or its image cache.

## 4. Publish the artifact to an OCI registry

Publish the flattened artifact under a normal OCI registry reference. Use a versioned tag while
iterating, then record the registry digest after publication. Prefer the digest when repeatability
matters because a tag can move.

The future publishing interface is expected to:

- Authenticate with the selected OCI-compatible registry.
- Upload the complete flattened artifact and its required metadata.
- Return or display a pullable reference and immutable digest.
- Fail clearly when the registry rejects credentials, storage, or artifact metadata.

This behavior is not yet available as a supported public SDK or CLI operation, so no executable
publishing example is shown here.

## 5. Load the published image locally

`Image.from_registry()` is the existing public constructor for an OCI registry reference:

```python
from cua import Image

image = Image.from_registry('REGISTRY/ORGANIZATION/IMAGE:TAG')
```

Replace the placeholder with the reference produced by the future publishing step. A digest-pinned
reference is preferred for a reproducible workflow.

After registry publication and local boot support are completed, pass this `Image` to the normal
local sandbox creation flow. The exact sandbox call depends on whether you want a persistent or
ephemeral sandbox; see [Sandbox lifecycle](/how-to-guides/sandbox/lifecycle) for the supported lifecycle
methods.

**Expected behavior:** the local runtime pulls the artifact, caches it, recognizes it as a Linux
sandbox image, and boots the same tested environment without requiring the original image definition
or build directory.

## Portability contract

The completed workflow should preserve these guarantees:

- **Self-contained:** the published artifact has no dependency on build-machine-only disk layers or
  caches.
- **Content-addressable:** the registry reports an immutable digest for the published content.
- **Reproducible consumption:** the same digest resolves to the same sandbox environment.
- **Local reuse:** `Image.from_registry()` can represent the artifact for local sandbox creation.
- **Clear compatibility errors:** unsupported artifact formats, operating systems, or host runtimes
  fail before an unusable sandbox is returned.

Until the missing publishing and local boot integration is implemented and documented as supported,
use this page as the public design promise for the portable workflow rather than an operational
runbook.
