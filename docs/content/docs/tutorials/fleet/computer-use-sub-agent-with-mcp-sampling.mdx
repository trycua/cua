# Build a computer-use sub-agent with MCP Sampling

This tutorial describes a safe control boundary for a computer-use sub-agent. The host MCP client owns
model-provider access and user approval. A custom sub-agent MCP server runs the agent loop and uses a
claimed Fleet Cua Driver endpoint as its downstream computer interface.

## Learning outcome

You can design a bounded MCP Sampling loop in which the host approves model requests, the custom
sub-agent server performs downstream Cua Driver actions, and Fleet claims are released after the task.

## Prerequisites

- Fleet access, per-user Fleet credentials, and Python 3.12 or 3.13 for the claim workflow.
- A Fleet pool declaring `cua-driver`, plus a claimed desktop whose proxied MCP URL has been verified
  through [Connect to a Fleet service](/how-to-guides/fleet/connect-to-a-service).
- A host MCP client with model-provider credentials that supports MCP Sampling. Not every MCP client
  supports Sampling.
- The [MCP Sampling specification](https://modelcontextprotocol.io/specification/2025-11-25/client/sampling),
  which is the protocol authority for this tutorial.

<Callout type="info">
  Cua Driver does not emit Sampling requests. Your custom sub-agent server owns the Sampling loop
  and calls Cua Driver tools as its downstream computer interface.
</Callout>

## 1. Define the control boundary

Keep each responsibility on the side that can safely enforce it:

| Component                   | Responsibility                                                                                                     |
| --------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| Host MCP client             | Owns the model account, presents approval/edit/deny controls, and sends model requests.                            |
| Custom sub-agent MCP server | Handles the originating `tools/call`, requests Sampling, bounds the loop, and invokes downstream Cua Driver tools. |
| Fleet Cua Driver            | Provides the desktop inspection and action tools through the authenticated claimed-service endpoint.               |

The host must declare Sampling during initialization. When the sub-agent sends tool-enabled Sampling
requests, the host must also have declared `sampling.tools`; a server must not send tool-enabled Sampling
requests to a client that did not declare that capability.

```json
{
  "capabilities": {
    "sampling": {
      "tools": {}
    }
  }
}
```

## 2. Hold and connect the downstream desktop

Reuse the session-holder procedure in
[Connect Hermes to Fleet Cua Driver](/tutorials/fleet/hermes-and-cua-driver). Start it before the
sub-agent session and leave it running until cleanup. It creates and verifies the claim, checks
`/healthz`, and writes these private environment variables without printing the bearer token:

```text
FLEET_CUA_DRIVER_URL
FLEET_ACCESS_TOKEN
```

Start the trusted custom server from a shell that sourced that private environment file. The custom
server, not the model host, uses the authenticated endpoint to inspect and control the desktop. In a
production deployment, the server must own or supervise the equivalent claim holder and release the
claim in its own `finally` path.

## 3. Associate Sampling with an active tool call

The host begins the work by calling a tool exposed by the custom sub-agent server, for example
`run_computer_task`. Keep that originating `tools/call` request open while the server asks the host for
Sampling. This ties the model requests to a user-visible request, lets the host apply its approval
policy, and gives the server a clear cancellation boundary.

For every Sampling request, send the task, planned tool interface, and relevant desktop observations to
the host. The MCP client can present review, approval, editing, or denial before it returns a Sampling
result or error. The server must handle that result or error and return an informative tool result to
the originating call when it cannot continue.

## 4. Follow the Sampling tool loop

The following is an exact JSON-RPC message flow for one inspection turn. It uses a conceptual
`inspect_desktop` tool exposed by the custom server to the model; after approval, the custom server maps
that request to an actual Cua Driver state-inspection call. Each block is valid JSON.

**Server to host: Sampling request**

```json
{
  "jsonrpc": "2.0",
  "id": 41,
  "method": "sampling/createMessage",
  "params": {
    "messages": [
      {
        "role": "user",
        "content": {
          "type": "text",
          "text": "Inspect the browser and determine whether https://example.com is visible."
        }
      }
    ],
    "tools": [
      {
        "name": "inspect_desktop",
        "description": "Return current desktop or browser-window state from the claimed Fleet desktop.",
        "inputSchema": {
          "type": "object",
          "properties": {
            "target": {
              "type": "string",
              "enum": ["desktop", "browser-window"]
            }
          },
          "required": ["target"]
        }
      }
    ],
    "toolChoice": {
      "mode": "required"
    },
    "maxTokens": 500
  }
}
```

**Host to server: approved model response**

```json
{
  "jsonrpc": "2.0",
  "id": 41,
  "result": {
    "role": "assistant",
    "content": [
      {
        "type": "tool_use",
        "id": "inspect-1",
        "name": "inspect_desktop",
        "input": {
          "target": "browser-window"
        }
      }
    ],
    "model": "host-selected-model",
    "stopReason": "toolUse"
  }
}
```

The custom server now calls the relevant Cua Driver inspection tool through the claimed endpoint and
records the result. It must return a tool result with the matching tool-use ID in the next Sampling
request; a user message containing tool results must contain only tool results.

**Server to host: continuation with downstream result**

```json
{
  "jsonrpc": "2.0",
  "id": 42,
  "method": "sampling/createMessage",
  "params": {
    "messages": [
      {
        "role": "user",
        "content": {
          "type": "text",
          "text": "Inspect the browser and determine whether https://example.com is visible."
        }
      },
      {
        "role": "assistant",
        "content": [
          {
            "type": "tool_use",
            "id": "inspect-1",
            "name": "inspect_desktop",
            "input": {
              "target": "browser-window"
            }
          }
        ]
      },
      {
        "role": "user",
        "content": [
          {
            "type": "tool_result",
            "toolUseId": "inspect-1",
            "content": [
              {
                "type": "text",
                "text": "Observed browser state: URL https://example.com/; heading Example Domain; screenshot captured."
              }
            ]
          }
        ]
      }
    ],
    "maxTokens": 500
  }
}
```

## 5. Implement the server loop

No current repository SDK provides a verified runnable Sampling handler, so the following is
framework-neutral pseudocode, not copy-paste library code:

```text
on tools/call run_computer_task(task):
    require the host client to have declared sampling
    require sampling.tools before sending a request with tools
    require that the Fleet claim is Bound and the endpoint belongs to that claim
    keep the originating request active and set an overall deadline

    for iteration in 1..MAX_ITERATIONS while before deadline:
        request = build_sampling_request(history, computer_tool_definitions)
        await sampling/createMessage(request)
        if the MCP client returns an error or cancellation:
            return an informative tool result for the originating request

        append the returned model response to history
        if model_response.stopReason is endTurn:
            return the final tool result for the originating request

        for each requested computer tool use:
            enforce the server-side action allowlist and argument validation
            enforce per-action and overall time limits
            result = call the corresponding Cua Driver tool downstream
            append a matching tool_result to history

    return a bounded-loop or deadline-exceeded tool result
finally:
    release the claim when this server owns it, or stop its supervised claim holder
```

Use a small, explicit `MAX_ITERATIONS`. On the last permitted turn, either stop with a bounded-loop
result or request a final answer without additional tools. The server itself must enforce claim binding,
action allowlists, iteration and time limits, and cleanup. Return downstream observations to the next
Sampling message rather than relying on the model's memory; host review does not replace those controls.

## 6. Verify and clean up

Use Cua Driver state and screenshots to verify the desktop result, rather than relying on the sub-agent's
summary alone. These observations show the final visible state; they do not prove that a trusted server
made no other downstream calls. Release the claim through the server-owned cleanup path, including on
denial, cancellation, and iteration-limit exits. For the tutorial session holder, follow
[Connect Hermes to Fleet Cua Driver](/tutorials/fleet/hermes-and-cua-driver).

<Callout type="warn">
  Do not send credentials, secrets, or sensitive desktop content to a model unless the host's
  approval and data-handling policy explicitly permits it.
</Callout>

## What happened

The host retained model access and could review Sampling requests. The trusted custom server enforced its
own Fleet claim and action controls, translated returned model tool uses into Cua Driver calls, and fed
the resulting observations back through Sampling. Cua Driver remained a downstream computer interface
rather than a Sampling participant.
