# Benchmark Codex and Claude Code with Cua Bench

This tutorial defines a small, repeatable computer-use comparison between Codex and Claude Code. It
uses Fleet for isolated desktops and Cua Bench for task setup, evaluation, and artifacts.

<Callout type="warn">
  Important current boundary: this checkout does not contain a built-in Cua Bench `codex` or
  `claude-code` agent class. Do not invent either name. You must register two custom adapters that
  implement `cua_bench.agents.BaseAgent.perform_task(...)` and drive the supplied `DesktopSession`.
</Callout>

## Learning outcome

You can run repeated, like-for-like trials of two external coding agents, preserve the evidence Cua
Bench actually records, and report results without mistaking Fleet or Cua Bench implementation gaps
for model differences.

## Prerequisites

- Fleet access and the credentials from [Use the Fleet SDK workflow](/how-to-guides/fleet/use-the-sdk).
- A Fleet pool with a fixed Linux desktop image digest, fixed CPU and memory requests, and all tools
  needed by both adapters. Treat the digest, not a mutable tag, as the environment identity; see
  [Fleet images](/concepts/fleet/images).
- A source checkout with `libs/cua-bench`, a working `cb` command, and the external Codex and Claude
  Code installations and credentials that your adapters will use.
- A Fleet-backed Cua Bench session/provider adapter. This checkout's Cua Bench desktop factory only
  recognizes `simulated` and `native`, and its `CloudProvider` targets CUA Cloud Evals, not Fleet.
  In particular, do not call `--provider cloud` or `CloudProvider` a Fleet integration.

## 1. Freeze the comparison contract

Before launching either agent, record one shared contract. Make both agents consume the same Fleet
pool revision: same image digest, CPU and memory, desktop resolution, task text, initial state,
maximum Cua Bench steps, external-agent wall-clock timeout, evaluator, and number of trials.

The following is an **adapter contract, not a Cua Bench configuration file**. The current checkout
has no Fleet provider field or parser flag for these values. Your Fleet adapter must write a copy next
to every run and reject a trial whose values do not match the baseline.

```yaml
fleet_pool: bench-linux
fleet_image_digest: registry.example/bench-linux@sha256:replace-with-pinned-digest
resources:
  cpu: '4'
  memory: 8Gi
desktop:
  width: 1024
  height: 768
task_id: save-three-lines-v1
max_steps: 20
external_agent_timeout_seconds: 900
initial_state: empty-Desktop-and-no-answer-file
evaluator: exact-Desktop-answer-file-v1
trials_per_agent: 5
```

The adapter boundary is verifiable:

1. Claim one fresh Fleet sandbox from that pool for each trial, wait for `Bound`, and record the
   claim and sandbox names without recording credentials.
2. Expose it as the `DesktopSession` supplied to `perform_task`. It must provide the operations the
   adapter uses, such as screenshots, actions, and guest commands.
3. Start from a fresh image-backed claim for every trial; do not reuse a modified desktop.
4. Enforce the 900-second external-agent timeout in the adapter or its supervisor, then close the
   session and delete the claim in a `finally` path.
5. Persist the contract, Cua Bench run ID, claim lifecycle, final screenshot, and evaluator result
   into that trial's output directory.

Fleet supplies parallel, isolated claims after this boundary exists. It is not supplied by Cua Bench
in this checkout.

## 2. Define a deterministic task

Create `tasks/save_three_lines/main.py`. It uses the source-backed `native` task provider only as a
local Cua Bench task definition. `native` is not Fleet. Your Fleet session/provider integration must
replace the standard `get_session("native")` path before it can run this task on Fleet.

The task asks the agent to create a visible Desktop document. Setup removes any previous result;
evaluation checks the saved guest state rather than the agent's self-report. The exact expected text
is in task metadata, not an accessible reference file in the desktop, so the evaluator does not leak
a reward target into the initial state.

```python
import base64
import shlex

import cua_bench as cb


EXPECTED_TEXT = "alpha\nbravo\ncharlie\n"
TASK_DESCRIPTION = (
    "Using the visible desktop text editor, create ~/Desktop/answer.txt containing exactly three "
    "lines: alpha, bravo, and charlie. Save the document, leave it on the Desktop, and do not "
    "change any other files."
)


@cb.tasks_config(split="train")
def load_tasks():
    return [
        cb.Task(
            task_id="save-three-lines-v1",
            description=TASK_DESCRIPTION,
            metadata={"expected_text": EXPECTED_TEXT},
            computer={
                "provider": "native",
                "setup_config": {"os_type": "linux", "width": 1024, "height": 768},
            },
        )
    ]


@cb.setup_task(split="train")
async def set_up_task(task_cfg: cb.Task, session: cb.DesktopSession):
    await session.run_command("mkdir -p ~/Desktop && rm -f ~/Desktop/answer.txt")


@cb.evaluate_task(split="train")
async def evaluate_task(task_cfg: cb.Task, session: cb.DesktopSession) -> list[float]:
    expected = base64.b64encode(task_cfg.metadata["expected_text"].encode()).decode("ascii")
    script = (
        "import base64, pathlib, sys; "
        "expected = base64.b64decode(sys.argv[1]); "
        "path = pathlib.Path.home() / 'Desktop' / 'answer.txt'; "
        "actual = path.read_bytes() if path.exists() else b''; "
        "raise SystemExit(0 if actual == expected else 1)"
    )
    command = f"python3 -c {shlex.quote(script)} {shlex.quote(expected)}"
    result = await session.run_command(command, check=False)
    return [1.0 if result["return_code"] == 0 else 0.0]
```

For a held-out smoke check, duplicate the task under a different directory and change both the
description and metadata to a different three-line value. Keep that directory out of adapter
development and prompt tuning. The evaluator is deterministic only if the Desktop starts clean and
the image and task module stay pinned. `cb run task` has no split flag, so its documented command
cannot select `split="test"`; use the source-level held-out loop in the reinforcement-learning
tutorial for an adapter-driven test split.

## 3. Register custom adapters

`BaseAgent` currently requires this exact method signature:

```python
async def perform_task(
    self,
    task_description: str,
    session: "DesktopSession",
    logging_dir: Path | None = None,
    tracer=None,
) -> AgentResult:
    ...
```

Start each adapter with that contract. These class names are local examples, not built-in Cua Bench
agents. A complete adapter should render the instruction, bridge the external agent to the supplied
session, copy only safe logs to `logging_dir`, return token counts only when it has measured them,
and map failures to `FailureMode`.

```python
from pathlib import Path

from cua_bench.agents.base import AgentResult, BaseAgent


class CodexAdapter(BaseAgent):
    @staticmethod
    def name() -> str:
        return "codex-custom"

    async def perform_task(
        self,
        task_description: str,
        session: "DesktopSession",
        logging_dir: Path | None = None,
        tracer=None,
    ) -> AgentResult:
        raise NotImplementedError("Bridge Codex to the supplied DesktopSession here.")


class ClaudeCodeAdapter(BaseAgent):
    @staticmethod
    def name() -> str:
        return "claude-code-custom"

    async def perform_task(
        self,
        task_description: str,
        session: "DesktopSession",
        logging_dir: Path | None = None,
        tracer=None,
    ) -> AgentResult:
        raise NotImplementedError("Bridge Claude Code to the supplied DesktopSession here.")
```

The external command protocols are deliberately not shown as runnable commands here: their CLI and
noninteractive contracts can change. The required bridge behavior is:

```text
PSEUDOCODE - not runnable

for the selected external coding agent:
  action_count = 0
  before every computer action exposed to the external agent:
    if action_count >= ACTION_BUDGET:
      stop and return AgentResult(failure_mode=FailureMode.MAX_STEPS_EXCEEDED)
    issue exactly one supplied DesktopSession action
    action_count += 1
  pass the rendered task description and only these guarded desktop-action tools to the agent
  enforce the same wall-clock timeout, record measured token totals when available, and write logs
  return AgentResult(token totals, failure_mode) after the evaluator can run
```

`ACTION_BUDGET` must equal the shared contract's `max_steps` value. `--max-steps` is an additional
Cua Bench harness limit, not a sufficient substitute: an external adapter can call `DesktopSession`
directly. Count every computer action the adapter issues through its bridge, stop before action 21 in
this example, and return `FailureMode.MAX_STEPS_EXCEEDED` for both agents under the same rule.

Register the adapters using the source-backed `.cua/agents.yaml` `import_path` mechanism. The module
must be importable in the environment where Cua Bench launches the agent.

```yaml
agents:
  - name: codex-custom
    import_path: bench_agents:CodexAdapter
    defaults:
      model: codex-external-model-id
  - name: claude-code-custom
    import_path: bench_agents:ClaudeCodeAdapter
    defaults:
      model: claude-code-external-model-id
```

## 4. Run repeated trials

`cb run task` accepts `--agent`, `--model`, `--max-steps`, and `--output-dir`; it does **not** expose
a timeout flag in the current parser. The adapter contract from step 1 therefore owns the timeout.
Use `--wait` so each command returns only after the trial finishes, and use a separate output
directory for every agent/trial pair.

```bash
for trial in 01 02 03 04 05; do
  cb run task ./tasks/save_three_lines \
    --variant-id 0 \
    --wait \
    --agent codex-custom \
    --model "$CODEX_MODEL" \
    --max-steps 20 \
    --output-dir "./artifacts/codex/trial-${trial}"
done

for trial in 01 02 03 04 05; do
  cb run task ./tasks/save_three_lines \
    --variant-id 0 \
    --wait \
    --agent claude-code-custom \
    --model "$CLAUDE_CODE_MODEL" \
    --max-steps 20 \
    --output-dir "./artifacts/claude-code/trial-${trial}"
done
```

Do not compare a local `native` run with a Fleet-backed run. The commands above are valid Cua Bench
syntax, but they use Fleet only after your provider/session adapter intercepts provisioning and
records the same contract for every invocation.

## 5. Verify and compare evidence

For every trial, independently confirm that the evaluator reward is `1.0`, `answer.txt` is visible
on the final Desktop, and the final screenshot matches the saved document. Compare:

1. Success rate and mean reward across all trials.
2. End-to-end latency, measured from claim request through evaluation, plus the Cua Bench duration if
   your runner records it.
3. Failure modes, including `api_error` and `max_steps_exceeded`, and adapter or Fleet claim failures
   recorded separately from model failures.
4. Trajectories and screenshots only when the adapter/Cua Bench trace output captured them.
5. `AgentResult.total_input_tokens` and `total_output_tokens` only when the adapter populated those
   fields from the external agent. Cua Bench initializes them to zero; zero is not evidence of zero
   usage. Treat cost as external accounting, not a guaranteed harness output.

Five trials are a smoke comparison, not a ranking. Report the count, frozen contract, failures, and
per-trial outcomes; do not infer a reliable performance gap from a small sample.

## 6. Clean up intentionally

After each trial, the Fleet adapter should close its `DesktopSession` and delete the claim it created.
After the batch, list the claims created by the run label and delete any that remain, following the
`finally` cleanup path in [Use the Fleet SDK workflow](/how-to-guides/fleet/use-the-sdk). Stop any
Cua Bench sessions that did not reach a terminal state with `cb run stop <run-id>`.

Retain the per-trial contracts, evaluator result, run IDs, logs, and screenshots needed for the
comparison. Delete temporary desktop credentials, local session metadata, and artifacts that contain
unnecessary task data or external-agent output.

## What happened

Cua Bench supplied the task decorators, evaluator, agent interface, and run artifact layout. Fleet
supplied fresh isolated desktops only through an explicit adapter, while the two custom adapters made
the external coding agents comparable without pretending they are built into Cua Bench.
