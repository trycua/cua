# `@trycua/cyclops` TypeScript SDK Implementation Plan

## 1. Objective and Source Findings

Create a public Node.js-first TypeScript SDK named `@trycua/cyclops` in the `trycua/cua` monorepo. The SDK authenticates with a Cyclops user key, manages namespaces and user keys, creates and claims OSGym pools, waits for lifecycle transitions, and exposes authenticated sandbox service and MCP connection helpers.

The plan is based on these current implementation patterns:

- `trycua/cloud/cyclops-cs/src/api/generated/cyclops-cs-backend.ts` is generated by `swagger-typescript-api`. Its `HttpClient` defaults to `baseUrl: "/"`, browser `fetch`, `credentials: "same-origin"`, and per-request `securityWorker` injection. It is useful for documenting backend CRUD routes, but its defaults and generated response wrapper are SPA-oriented.
- `trycua/cloud/cyclops-cs/src/api/client.ts` creates one generated `Api` singleton with `baseUrl: ""` and a `securityWorker` that calls `getToken()` before each secure request and injects `Authorization: Bearer ...`.
- `trycua/cloud/cyclops-cs/src/auth/keycloak.ts` uses browser authorization-code/PKCE through `keycloak-js`, refreshes when fewer than 30 seconds remain, and centralizes token retrieval in `getToken()`.
- `trycua/cloud/cyclops-cs/src/api/cyclops.ts` hand-writes the Kubernetes proxy calls because pools and claims are custom resources not represented in the backend OpenAPI surface. It uses a small `fetchJson` helper, `/api/k8s` for Kubernetes resources, `/api/svc` for sandbox services, and polling in the Python examples for warm/bound/ready transitions.
- `trycua/cloud/cyclops-cs/src/api/keys.ts` keeps a stable hand-written facade around generated key-management methods rather than exposing generated names and response wrappers directly.
- `trycua/cua/libs/typescript/core` establishes the package pattern: one directory per package under `libs/typescript`, ESM, `tsdown`, `vitest`, strict TypeScript, public npm publishing, and build-time `__CUA_VERSION__` injection. `cuaVersionHeaders("cyclops", version)` should be applied to Cyclops API requests.
- `trycua/cua/libs/python/core/cua_core/http.py` confirms the cross-language `X-Cua-Client-Version` convention.
- The existing Cyclops Python convenience client exchanges `client_credentials`, caches the token deadline, refreshes 30 seconds early, and uses a lock to collapse concurrent refreshes. The TypeScript implementation should preserve those semantics with a shared in-flight promise.

## 3. Public API Design

### 3.1 Primary constructor

The required ergonomic API is:

```ts
import { CyclopsClient } from '@trycua/cyclops';

const cyclops = new CyclopsClient(
  'https://run.cua.ai',
  process.env.CUA_CLIENT_ID!,
  process.env.CUA_CLIENT_SECRET!,
);
```

Implement overloads so the positional constructor remains exactly available while an options object supports advanced configuration:

```ts
export class CyclopsClient {
  constructor(baseUrl: string, clientId: string, clientSecret: string);
  constructor(options: CyclopsClientOptions);
}

export interface CyclopsClientOptions {
  baseUrl: string;
  clientId: string;
  clientSecret: string;
  tokenUrl?: string;
  fetch?: typeof fetch;
  timeoutMs?: number;
  refreshSkewMs?: number;
  headers?: HeadersInit;
  signal?: AbortSignal;
}
```

Defaults:

- `tokenUrl`: `https://auth.cua.ai/realms/cyclops-cs/protocol/openid-connect/token`.
- `timeoutMs`: 30 seconds for ordinary control-plane calls; wait helpers use their own total timeout.
- `refreshSkewMs`: 30 seconds.
- `fetch`: `globalThis.fetch`, requiring Node.js 18+ and avoiding a runtime HTTP dependency.
- Normalize `baseUrl` once by removing trailing slashes; construct every request with `new URL(path, baseUrl)` so Node always receives an absolute URL.

Recommended top-level methods and resource managers:

```ts
class CyclopsClient {
  readonly namespaces: NamespaceManager;
  readonly userKeys: UserKeyManager;

  createPool(spec: CreatePoolSpec): Promise<Pool>;
  getPool(name: string, namespace?: string): Promise<Pool>;
  deletePool(name: string, namespace?: string): Promise<void>;
  listPools(options?: ListPoolsOptions): Promise<Pool[]>;

  getClaim(namespace: string, name: string): Promise<Claim>;
  request<T>(path: string, init?: CyclopsRequestInit): Promise<T>;
  close(): void;
}
```

The positional pool methods requested by the SDK brief may also be supported as compatibility overloads:

```ts
createPool(name: string, spec: PoolSpec): Promise<Pool>;
getPool(namespace: string, name: string): Promise<Pool>;
deletePool(namespace: string, name: string): Promise<void>;
```

Prefer the object form in documentation because it is extensible and prevents argument-order mistakes.

### 3.2 Resource handles

Return resource objects rather than raw Kubernetes JSON. Handles retain the client and identity, expose the latest snapshot, and return new refreshed handles from lifecycle methods.

```ts
export class Pool {
  readonly name: string;
  readonly namespace: string;
  readonly resource: OSGymWorkspacePool;

  refresh(): Promise<Pool>;
  delete(): Promise<void>;
  claim(spec?: ClaimSpec): Promise<Claim>;
  waitReady(options?: WaitOptions): Promise<Pool>;
  waitWarm(options?: WaitWarmOptions): Promise<Pool>;
}

export class Claim {
  readonly name: string;
  readonly namespace: string;
  readonly resource: OSGymSandboxClaim;

  refresh(): Promise<Claim>;
  waitBound(options?: WaitOptions): Promise<Sandbox>;
  release(): Promise<void>;
}

export class Sandbox {
  readonly name: string;
  readonly namespace: string;
  readonly claim: Claim;

  serviceUrl(serverLabel: string, path?: string): URL;
  mcpUrl(serverLabel: string, path?: string): URL;
  waitReady(serverLabel: string, options?: ServiceWaitOptions): Promise<void>;
  fetch(serverLabel: string, path: string, init?: RequestInit): Promise<Response>;
  mcp(serverLabel: string, options?: McpConnectionOptions): McpEndpoint;
}
```

Required behavior:

- `pool.claim()` creates `OSGymSandboxClaim` in the pool namespace and defaults `sandboxTemplateRef.name` to `${pool.name}-template`.
- `pool.waitReady()` waits for the compatibility `OSGymWorkspacePool.status.phase` to reach a ready value. Treat absent/unknown phases as not ready; fail immediately for terminal failure phases when present.
- `pool.waitWarm()` waits until `status.availableCount >= minAvailable` (default 1). This is the meaningful readiness gate before a claim can bind quickly.
- `claim.waitBound()` polls until `status.phase === "Bound"` and `status.sandbox.name` exists. Throw a typed lifecycle error on `Failed`; include the latest resource in timeout/failure errors.
- `claim.release()` deletes the claim. Deletion is the release operation: the operator resets/recycles or garbage-collects the bound sandbox according to pool policy.
- `sandbox.mcpUrl(serverLabel)` builds an absolute backend proxy URL of the form `${baseUrl}/api/svc/${namespace}/${sandboxName}-${serverLabel}/mcp`. It must not return the internal cluster DNS URL because an external Node SDK cannot resolve `*.svc.cluster.local`.

### 3.3 Namespace and key management

```ts
export interface NamespaceManager {
  list(): Promise<CyclopsNamespace[]>;
  create(name: string): Promise<CyclopsNamespace>;
  get(name: string): Promise<CyclopsNamespace | undefined>;
  delete(name: string): Promise<void>;
}

export interface UserKeyManager {
  list(): Promise<UserKey[]>;
  create(input: CreateUserKeyInput): Promise<CreatedUserKey>;
  delete(id: string): Promise<void>;
}

export interface CreateUserKeyInput {
  name: string;
  scope?: string[];
}
```

Important auth distinction to document and validate:

- User keys (`ukey-...`, created by `/api/user-keys`) carry the owner's identity and can use `/api/k8s` to manage namespaces, pools, and claims.
- Per-pool keys (`key-...`, created by `/api/keys`) are for pool-scoped service/API access and do not have Kubernetes identity. They must not be presented as sufficient credentials for pool/claim lifecycle methods.
- The SDK constructor accepts any credential syntactically, because server policy is authoritative, but when `clientId` begins with `key-`, lifecycle methods should throw an early `CyclopsCredentialScopeError` with a clear instruction to use a user key. Service proxy helpers may still be usable with a pool key where backend ownership rules allow it.

User-key creation returns the secret once. Model that explicitly and never log it:

```ts
export interface UserKey {
  id: string;
  clientId: string;
  name: string;
  scope: string[];
}

export interface CreatedUserKey {
  clientId: string;
  clientSecret: string;
  tokenUrl: string;
  name: string;
  scope: string[];
}
```

## 4. Authentication and HTTP Transport

### 4.1 OAuth2 client credentials

Implement `ClientCredentialsTokenProvider`:

```ts
interface TokenResponse {
  access_token: string;
  token_type?: string;
  expires_in?: number;
  scope?: string;
}

class ClientCredentialsTokenProvider {
  getToken(options?: { forceRefresh?: boolean }): Promise<string>;
  invalidate(token?: string): void;
}
```

Exchange requirements:

- POST the token URL using `application/x-www-form-urlencoded`.
- Body fields: `grant_type=client_credentials`, `client_id`, `client_secret`.
- Require a non-empty `access_token`.
- Use `expires_in`, defaulting defensively to 300 seconds if omitted.
- Cache `expiresAt = Date.now() + expires_in * 1000`.
- Refresh when `Date.now() >= expiresAt - refreshSkewMs`.
- Collapse concurrent refreshes with one `refreshPromise`; clear it in `finally`.
- Do not use browser `keycloak-js`, refresh tokens, cookies, local storage, or PKCE. Client credentials normally re-mint access tokens with the client secret.
- Redact `clientSecret`, access tokens, and token endpoint response bodies from error messages.

### 4.2 Request pipeline

Implement one `HttpClient`/transport shared by all managers:

```ts
interface CyclopsTransport {
  request<T>(path: string, init?: CyclopsRequestInit): Promise<T>;
  raw(path: string, init?: CyclopsRequestInit): Promise<Response>;
}
```

Every authenticated request should:

1. Resolve an absolute URL from the normalized base URL.
2. Obtain a fresh-enough bearer token.
3. Merge headers case-insensitively, allowing explicit request headers without dropping auth/version headers.
4. Send `Authorization: Bearer <token>`.
5. Send `X-Cua-Client-Version: cyclops:<package version>` via `cuaVersionHeaders('cyclops', __CUA_VERSION__)` from `@trycua/core`.
6. Set `Accept: application/json` by default.
7. Set `Content-Type: application/json` only when serializing a JSON body; token exchange uses form encoding and MCP may set its own headers.
8. Combine caller cancellation with a timeout `AbortSignal`.
9. Parse JSON by content type, tolerate `204`/empty success bodies, and expose raw `Response` for streaming/MCP/service calls.
10. On `401`, invalidate the token and retry exactly once with a forced refresh. Do not retry arbitrary non-idempotent failures.

Errors:

```ts
class CyclopsError extends Error {}
class CyclopsHttpError extends CyclopsError {
  status: number;
  method: string;
  url: string;
  requestId?: string;
  body?: unknown;
}
class CyclopsAuthError extends CyclopsError {}
class CyclopsTimeoutError extends CyclopsError {}
class CyclopsLifecycleError extends CyclopsError {}
class CyclopsCredentialScopeError extends CyclopsError {}
```

Bound error bodies to a safe size and parse structured `{ error }` responses where available.

## 5. Code Generation Decision

### Decision: hand-write a thin public client and share only schemas/conformance inputs, not the SPA-generated runtime

Do not directly reuse `cyclops-cs/src/api/generated/cyclops-cs-backend.ts` as the SDK runtime.

Rationale:

- It is generated inside the cloud repository, not the public SDK monorepo; sharing it creates a cross-repository release/version dependency.
- Its defaults are browser-specific: relative `/api/...` paths, `baseUrl: "/"`, `credentials: "same-origin"`, global browser-oriented `fetch`, and cancellation-token machinery not needed by the SDK.
- The SPA singleton intentionally uses `baseUrl: ""` so nginx resolves same-origin paths. A Node client must construct absolute URLs from `baseUrl`.
- The SPA auth worker calls browser Keycloak PKCE token refresh. The SDK needs OAuth2 `client_credentials` and secret-safe refresh behavior.
- The core SDK surface (pool, claim, sandbox service/MCP) is not represented by ordinary generated backend operations. Pools and claims are hand-written Kubernetes proxy calls today, so generation would still require a second abstraction layer.
- Exposing generated names such as `namespacesList` and `HttpResponse.data` would leak backend generator details into the stable public API.

Do not generate a second full `swagger-typescript-api` runtime for v1. That would duplicate a large browser-shaped client while still leaving the most important K8s proxy APIs hand-written.

Instead:

- Hand-write a small Node-first transport and stable domain managers.
- Hand-write the small set of backend route DTOs for namespaces and user keys.
- Define typed Kubernetes resource shapes for pools/claims from the CRD schema.
- Add contract tests against checked-in fixtures extracted from the backend Swagger/CRDs.
- Optionally add a future internal generation script that emits type-only DTOs from `swagger.json`/CRDs, but generated files must remain internal and must not define the public API.

Revisit generation when Cyclops publishes a consolidated, versioned OpenAPI document that includes pool, claim, wait, and service proxy operations as first-class endpoints. At that point, generated transport code can sit under `src/generated/`, wrapped by the same stable domain API.

- Preserve unknown status/condition fields; do not over-constrain server evolution.

## 7. Kubernetes and Backend Route Mapping

Centralize route creation in `src/routes.ts`; never concatenate unescaped user input elsewhere.

```text
POST   /api/namespaces
GET    /api/namespaces
DELETE /api/namespaces/{name}

GET    /api/user-keys
POST   /api/user-keys
DELETE /api/user-keys/{id}

POST   /api/k8s/apis/cua.ai/v1/namespaces/{ns}/osgymworkspacepools
GET    /api/k8s/apis/cua.ai/v1/namespaces/{ns}/osgymworkspacepools/{name}
DELETE /api/k8s/apis/cua.ai/v1/namespaces/{ns}/osgymworkspacepools/{name}

POST   /api/k8s/apis/osgym.cua.ai/v1alpha1/namespaces/{ns}/osgymsandboxclaims
GET    /api/k8s/apis/osgym.cua.ai/v1alpha1/namespaces/{ns}/osgymsandboxclaims/{name}
DELETE /api/k8s/apis/osgym.cua.ai/v1alpha1/namespaces/{ns}/osgymsandboxclaims/{name}

ANY    /api/svc/{namespace}/{sandbox}-{serviceLabel}/{path}
```

`createPool` flow:

1. Resolve `namespace = input.namespace ?? input.name` to preserve the SPA's 1:1 default while permitting explicit future namespaces.
2. If `createNamespace !== false`, call namespace create first.
3. Ignore only a typed HTTP 409 from namespace creation; propagate every other error.
4. POST an `OSGymWorkspacePool` with `metadata.name`, `metadata.labels['cua.ai/pool']`, and the supplied spec.
5. Return the server resource as a `Pool` handle. If the proxy returns no body, immediately GET it before returning.
6. Do not automatically roll back a pre-existing namespace after a pool failure. For a namespace newly created by this call, make cleanup opt-in in a later version rather than risking deletion of user resources.

Claim names default to `claim-${crypto.randomUUID().slice(0, 8)}`. Use the Web Crypto API available in modern Node rather than another UUID dependency.

## 8. Wait and Polling Semantics

Use a shared abortable poller:

```ts
export interface WaitOptions {
  timeoutMs?: number;
  intervalMs?: number;
  signal?: AbortSignal;
}

export interface WaitWarmOptions extends WaitOptions {
  minAvailable?: number;
}
```

Defaults:

- `pool.waitReady`: timeout 10 minutes, interval 2 seconds.
- `pool.waitWarm`: timeout 20 minutes, interval 5 seconds because cold image pulls and Windows boot can be slow.
- `claim.waitBound`: timeout 5 minutes, interval 2 seconds.
- `sandbox.waitReady`: timeout 7 minutes, interval 5 seconds; retry proxy `502`, `503`, and `504` because the claim may be bound while the guest is rebooting/resetting.

Implementation rules:

- Check the condition immediately before the first sleep.
- Respect caller aborts and total deadlines.
- Add bounded jitter (for example +/-10%) to prevent synchronized clients from polling in lockstep.
- Do not retry `401`, `403`, malformed data, or terminal lifecycle states.
- Treat `404` as retryable only where eventual creation/projection is expected and explicitly documented.
- Include last observed phase/status and elapsed time in `CyclopsTimeoutError`.
- Keep polling GETs idempotent; the generic transport should not globally retry POST/DELETE requests.

## 9. MCP Integration Helpers

### 9.1 URL and authenticated fetch helpers

At minimum, provide helpers that work without forcing a dependency on one MCP SDK version:

```ts
export interface McpEndpoint {
  url: URL;
  headers: Record<string, string>;
  fetch: typeof fetch;
}

sandbox.mcpUrl('mcp');
sandbox.mcp('mcp');
```

`McpEndpoint.fetch` must route through the Cyclops authenticated transport so every MCP request gets a current bearer token. A static token embedded once in `headers` is insufficient for long-lived MCP sessions. The returned `headers` field is a snapshot for short-lived/manual clients and documentation must recommend `fetch` for automatic refresh.

`mcpUrl(serverLabel, path = '/mcp')` behavior:

- Validate `serverLabel` as a service-name suffix.
- Build service name `${sandbox.name}-${serverLabel}`.
- Build the absolute proxy URL under `/api/svc`.
- Preserve the `/api/svc/{namespace}/{service}` prefix and append an encoded path safely.
- Never expose `clientSecret` in the URL or query string.

### 9.2 Optional official MCP SDK adapter

Avoid a mandatory `@modelcontextprotocol/sdk` dependency in the first release. Its API evolves independently and users may already have a chosen version. Provide an optional subpath adapter with a peer dependency if the repository's current dependency policy supports it:

```ts
import { createMcpClientTransport } from '@trycua/cyclops/mcp';

const transport = createMcpClientTransport(sandbox, 'mcp');
```

Proposed behavior:

- Adapter targets Streamable HTTP.
- It supplies the authenticated refresh-aware fetch implementation.
- It sets `Accept: application/json, text/event-stream` and leaves `Mcp-Session-Id` handling to the official SDK transport.
- Declare `@modelcontextprotocol/sdk` as an optional peer dependency, not bundled into the base package.

If an official transport cannot accept a custom fetch in the selected supported version, defer the adapter and ship only `McpEndpoint`; do not monkey-patch global fetch or freeze a bearer token.

### 9.3 Generic sandbox service calls

MCP is one service protocol. Expose generic methods for health checks and other APIs:

```ts
await sandbox.fetch('computer', '/status');
await sandbox.waitReady('mcp', { path: '/healthz' });
```

This mirrors the existing `/api/svc` examples and lets consumers wait for the guest service after `claim.waitBound()`.

## 12. Test Plan

Use Vitest and injected `fetch` stubs; no live production calls in unit tests.

### Auth tests

- Exchanges exact form fields and content type at the configured token URL.
- Uses the default token URL when omitted.
- Returns cached token before the refresh window.
- Refreshes 30 seconds early.
- Coalesces multiple concurrent callers into one token request.
- Falls back to a 300-second TTL when `expires_in` is absent.
- Rejects missing `access_token` with `CyclopsAuthError`.
- Redacts client secret/token response from errors.
- Force refresh and invalidation work after a 401.

Use Vitest fake timers for all expiry tests.

### Transport tests

- Converts relative API paths into correct absolute URLs for base URLs with/without trailing slash and with a deployment path prefix if supported.
- Injects bearer and `X-Cua-Client-Version` headers.
- Does not overwrite caller content type for form/MCP requests.
- Serializes JSON bodies and parses JSON responses.
- Handles 204 and empty 200 responses.
- Parses structured and text errors into `CyclopsHttpError`.
- Retries one 401 after invalidating the token, then stops.
- Honors timeout and caller abort signals.
- Preserves streaming/raw `Response` bodies for MCP.

### Pool tests

- `createPool` creates the namespace then posts the exact `OSGymWorkspacePool` body.
- Namespace 409 is tolerated; namespace 400/403/500 is not.
- `namespace ?? name` default matches the SPA convention.
- Explicit namespace and `createNamespace: false` work.
- Runtime, firmware, probes, services, OIDC, and autoscaling survive serialization.
- `getPool` and `deletePool` encode names safely.
- `waitReady` polls until ready and fails on terminal status.
- `waitWarm` waits for the configured `availableCount`.
- Pool-key credentials fail early for lifecycle methods.

### Claim tests

- `pool.claim()` defaults the template to `${pool.name}-template`.

- Generated claim name is DNS-safe.
- `waitBound` returns `Sandbox` only when phase is `Bound` and sandbox name exists.
- `Failed` produces `CyclopsLifecycleError` with the latest claim.

- `release` sends DELETE and tolerates an empty body.

- Generic service fetch receives a fresh bearer token.
- MCP fetch sends caller headers plus refreshed auth without leaking credentials in the URL.
- `waitReady` retries 502/503/504 and succeeds on any other response by default; optionally support an `accept(response)` predicate.

- Backend snake_case DTOs normalize to the public camelCase types.
- Missing optional arrays become empty arrays.

## 14. SPA Migration Path

The SPA can adopt the SDK for domain operations, but not by instantiating the required client-credentials constructor in browser code. Shipping a user key secret to a browser is unacceptable, and the SPA uses authorization-code/PKCE rather than client credentials.

Design the implementation so auth is injectable internally from day one:

```ts
export interface AccessTokenProvider {
  getToken(options?: { forceRefresh?: boolean }): Promise<string>;
  invalidate?(token?: string): void;
}
```

`CyclopsClient(baseUrl, clientId, clientSecret)` creates a `ClientCredentialsTokenProvider`. Also expose a safe advanced factory, initially marked as advanced but supported:

```ts
CyclopsClient.withTokenProvider({
  baseUrl: window.location.origin,
  tokenProvider: {
    getToken: () => getTokenFromKeycloak(),
  },
  fetch: window.fetch.bind(window),
});
```

Migration phases:

1. Implement the SDK in `trycua/cua` with no SPA changes; verify parity against fixtures and cloud examples.
2. Publish `@trycua/cyclops@0.x` and add it to `cyclops-cs` as a dependency.
3. Build a SPA adapter using the existing `getToken()` from `src/auth/keycloak.ts`, `baseUrl: window.location.origin` (or `""` normalized to origin by the adapter), and browser fetch.
4. Replace hand-written namespace and user-key modules first; these map directly to stable backend routes.
5. Replace pool and claim CRUD/wait logic next, preserving UI-specific view models outside the SDK.
6. Replace `/api/svc` URL assembly and MCP/service readiness helpers.
7. Keep admin dashboards, pod/node metrics, logs, restarts, Prometheus, and other operational K8s calls in the SPA until/unless they become intentional public SDK APIs.
8. Once all shared operations use the SDK, remove duplicate `fetchJson`, pool/claim DTOs, and route builders from the SPA.

Browser compatibility considerations:

- The core transport can be runtime-neutral if it uses standards APIs (`fetch`, `URL`, `Headers`, `AbortController`) and avoids Node-only modules in shared files.
- Keep client-credentials secret handling in a separate module/entry or ensure bundlers can tree-shake it. Documentation must state that the positional client-secret constructor is server-side only.
- Do not depend on `keycloak-js` in the SDK; the SPA owns its login flow and supplies tokens through the provider interface.
- The SDK should return domain resources, while the SPA retains presentation-specific summaries and React state.

Conclusion: yes, the SPA can adopt most of this SDK if token acquisition is abstracted. It should not use the client-secret constructor and should not migrate unrelated cluster-observability/admin endpoints into the public package by default.

## 16. Acceptance Criteria

The initial implementation is complete when:

- A Node 18+ consumer can construct `CyclopsClient(baseUrl, clientId, clientSecret)` and never manually manage bearer tokens.
- Concurrent requests trigger at most one refresh exchange and a backend 401 causes at most one forced-refresh retry.
- Pool create/get/delete, `pool.claim`, `pool.waitReady`, `pool.waitWarm`, `claim.waitBound`, and `claim.release` work through the documented Cyclops proxy routes.
- A bound sandbox exposes an absolute authenticated MCP proxy endpoint through `sandbox.mcpUrl(serverLabel)` and a refresh-aware fetch/helper suitable for Streamable HTTP MCP clients.
- Namespace and user-key CRUD are typed and secrets are not logged.
- Requests include `X-Cua-Client-Version: cyclops:<version>`.
- The package passes typecheck, Vitest, tsdown build, package-content smoke tests, and optional credentialed lifecycle integration testing.
- The public API does not expose `swagger-typescript-api` method names or browser Keycloak implementation details.
- The SPA can consume the same domain client through an injected Keycloak token provider without embedding a client secret.
