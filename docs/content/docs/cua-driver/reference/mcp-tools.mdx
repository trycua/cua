`cua-driver` exposes 34 MCP tools through a single stdio server (`cua-driver mcp`). Every tool is also callable from the shell as `cua-driver call <name> '<JSON-args>'`.

Tool names are `snake_case`. Responses are MCP `CallTool.Result` envelopes: a text block prefixed with a `✅` summary (or the error reason on failure), plus optional image or structured-content blocks on tools that produce them. See the [CLI reference](/cua-driver/reference/cli-reference) for shell flags like `--raw` and `--screenshot-out-file`.

<Callout type="info">
  Tool names match the CLI form exactly. `cua-driver call list_apps` and the MCP `list_apps` tool run the same code path.
</Callout>

<Callout type="info">
  **Two backends, one tool surface.** macOS ships the Swift driver by default (Accessibility / `CGEvent` / ScreenCaptureKit); Windows and Linux ship `cua-driver-rs` (UI Automation / `PostMessage` / ffmpeg). The tool names, arguments, and response shapes are the same on both. Descriptions below are synced from the Rust port and note the macOS-Swift behavior where it diverges. A few tools (`bring_to_front`, `kill_app`, `debug_window_info`, `get_accessibility_tree`) are Windows-first; the macOS Swift driver returns a pointer to the platform-native alternative.
</Callout>

<Callout type="info">
  **Daemon proxy.** When an MCP client spawns `cua-driver mcp` from an IDE terminal, the call may need to run under a different process identity than the spawning shell — on macOS to inherit `CuaDriver.app`'s TCC grants, on Windows to reach the interactive desktop from a Session 0 SSH login. `mcp` handles this by proxying to a running `cua-driver serve` daemon; tool semantics are unchanged. See the [process model](/cua-driver/guide/getting-started/process-model) (macOS) and [SSH on Windows](/cua-driver/guide/getting-started/windows-ssh) guides.
</Callout>

## Dispatch modes

Every pointer and keyboard tool on the Rust port takes an optional `dispatch` field controlling how the event is delivered:

- **`background`** (default) — never swaps the foreground window. Element-indexed actions and UIA-hit-tested pixel clicks deliver through the accessibility channel with no focus steal or visible flash. If a target's input stack would silently drop a backgrounded event (Chromium DOM content, GTK button widgets), the tool returns a structured `background_unavailable` error rather than failing silently.
- **`foreground`** — accepts a brief `SetForegroundWindow` swap (the `SendInput` path). Required to drive Chromium-content or GTK-widget targets reliably. Pair with [`bring_to_front`](#bring_to_front) to pay the foreground cost once for a sequence of calls instead of per call.
- **`auto`** — the driver's internal heuristics, with silent fallback to `SendInput` on known-problematic targets.

The macOS Swift driver keeps everything backgrounded by default and exposes the same field for parity.

## Permissions and discovery

### check_permissions

Check required permissions. On Windows this reports UI Automation availability, PostMessage access, and the daemon's integrity level; on macOS it reports the Accessibility and Screen Recording TCC grants and raises the system prompts for any that are missing.

**Arguments:** none.

### get_screen_size

Return the logical size of the main display in points plus its backing scale factor. Agents click in points; Retina displays report `scale_factor` 2.0. Requires no special permissions.

**Arguments:** none.

### get_cursor_position

Return the current mouse cursor position in screen points (origin top-left).

**Arguments:** none.

### list_apps

List apps — both currently running and installed-but-not-running — with per-app state flags: `running`, `active` (system-frontmost), `kind` (`"desktop"` or `"uwp"`), `launch_path` (what `launch_app` consumes), and `last_used`. Use it for "is X installed?" as well as "is X running?". For per-window state, call `list_windows` instead. Opening an app does not require this first — call `launch_app` directly.

**Arguments:** none.

### list_windows

List every top-level window known to the window manager. Each record self-contains its owning app identity, so callers never join back against `list_apps`. Per-record fields: `window_id` (HWND on Windows, CGWindowID on macOS), `pid` + `app_name`, `title`, `bounds`, `layer`, `z_index`, `is_on_screen`. The macOS-only `on_current_space` / `space_ids` fields are omitted on Windows.

**Arguments:**

- `pid` (integer, optional): Restrict the result to one pid's windows.
- `on_screen_only` (boolean, optional): Drop windows that aren't currently on-screen. Default false.

### get_accessibility_tree

Return a lightweight snapshot of the desktop: running processes and on-screen visible windows with their bounds and owner pid. This is a fast discovery read — for the full UIA subtree of a single window with clickable element indices, use [`get_window_state`](#get_window_state). Windows-first.

**Arguments:** none.

### get_window_state

Walk a running app's UIA tree (AX tree on macOS) and return a Markdown rendering of its UI, tagging every actionable element with `[element_index N]`. Pass those indices to `click`, `type_text`, `scroll`, etc. — those tools resolve the index to the cached element on the server.

INVARIANT: call `get_window_state` once per turn per `(pid, window_id)` before any element-indexed action against that window. The index map is replaced by the next snapshot of the same `(pid, window_id)`.

The tree walked is the window's tree (HWND-scoped); the screenshot and bounds come from the same `window_id`. `window_id` must belong to `pid`, or the call returns `isError: true` — the driver never picks a window implicitly. Set `query` to a case-insensitive substring to trim `tree_markdown` to matching lines plus their ancestor chain; element indices and `element_count` are unchanged.

`capture_mode` controls the response shape (default `som`):

- `som` — walks the tree **and** captures a screenshot. Element-indexed clicks work out of the box; the screenshot is there for disambiguation.
- `vision` — screenshot only; the tree walk is skipped and the element cache is not updated. Pair with pixel-addressed `click({pid, x, y})`.
- `ax` — tree only; no screenshot (no Screen Recording cost on macOS).

On Windows the Rust port has no standalone `screenshot` tool — the window screenshot is delivered inside `get_window_state` (`som` / `vision`). The macOS Swift driver still exposes a separate `screenshot` tool.

**Arguments:**

- `pid` (integer, required): Process ID from `list_apps`.
- `window_id` (integer, required): HWND of the target window. Must belong to `pid`. Enumerate via `list_windows` or read from `launch_app`'s `windows` array.
- `capture_mode` (string, optional): `som` (default), `vision`, or `ax`.
- `query` (string, optional): Case-insensitive substring filter on `tree_markdown`.

```json
{"pid":844,"window_id":10725}
```

### debug_window_info

Diagnostic: dump everything cua-driver sees about a pid's top-level windows from the daemon's session perspective — window class names, owning `.exe` basename + path, and (when UI Automation succeeds) the focused element with the list of patterns it supports (`ValuePattern`, `InvokePattern`, `TextPattern`, `TogglePattern`, …). Used to design and debug input routing for XAML / UWP / WinUI3 targets. Windows-only.

**Arguments:**

- `pid` (integer, required): PID of the process to inspect.

```json
{"pid":844}
```

## Pointer input

### click

Left-click against a target pid. **Prefer `element_index` over pixel coordinates** — it works on backgrounded / minimized / hidden / off-desktop windows, surfaces a stable handle, and tells you what you're clicking via the cached element's role + label. Reach for `x, y` only on a canvas / video / WebGL / custom-drawn surface that doesn't appear in the UIA tree.

- `element_index` + `window_id` (from the last `get_window_state` of that window) — performs the UIA Invoke pattern on the cached element via PostMessage. No cursor move, no focus steal. Requires a prior `get_window_state(pid, window_id)` this turn.
- `x`, `y` (window-local screenshot pixels) — on `dispatch:"background"` / `"auto"`, the driver first does a UIA hit-test at the resolved position: if the deepest invokable element exposes `InvokePattern`, it's invoked through the accessibility channel (flash-free, even on UWP / WinUI3 / Win11 packaged apps). The `PostMessage(WM_LBUTTONDOWN/UP)` fallback only runs when the hit-test misses, and returns `background_unavailable` if PostMessage is known to drop on that target — at which point switch to `dispatch:"foreground"`.

Exactly one of `element_index` or (`x` AND `y`) must be provided; `pid` is required in both. After a `zoom` call, pass `from_zoom=true` to translate zoom-image coords back to full-window space.

**Arguments:**

- `pid` (integer, required): Target process ID.
- `element_index` (integer, optional): Index from the last `get_window_state` for the same `(pid, window_id)`.
- `window_id` (integer, optional): HWND for the window that produced the `element_index`. Required when `element_index` is used.
- `x` (number, optional): X in window-local screenshot pixels. Provide together with `y`.
- `y` (number, optional): Y in window-local screenshot pixels. Provide together with `x`.
- `button` (string, optional): `left` (default), `right`, or `middle`. Windows-only convenience; macOS exposes right-click as a separate `right_click` tool.
- `count` (integer, optional): 1 (single), 2 (double), 3 (triple). Default 1.
- `dispatch` (string, optional): `background` (default), `foreground`, or `auto`. See [Dispatch modes](#dispatch-modes).
- `from_zoom` (boolean, optional): Coordinates are in the last `zoom` image for this pid.

```json
{"pid":844,"window_id":10725,"element_index":14}
```

### double_click

Double-click against a target pid. `element_index` + `window_id` synthesizes a stamped pixel double-click at the element's cached on-screen center; `x`, `y` posts two `WM_LBUTTONDOWN/UP` pairs to the deepest child window at that point. Exactly one addressing mode; `pid` required in both. The `modifier` field is accepted for macOS parity (no-op on Windows — PostMessage doesn't propagate modifier-key state).

**Arguments:**

- `pid` (integer, required): Target process ID.
- `element_index` (integer, optional): Index from the last `get_window_state`. Requires `window_id`.
- `window_id` (integer, optional): HWND for the target window.
- `x` / `y` (number, optional): Window-local screenshot pixels; provide together.
- `dispatch` (string, optional): See [Dispatch modes](#dispatch-modes).
- `from_zoom` (boolean, optional): Coordinates are in the last `zoom` image for this pid.
- `modifier` (array of string, optional): Accepted for parity; no-op on Windows.

```json
{"pid":844,"window_id":10725,"element_index":14}
```

### right_click

Right-click against a target pid. `element_index` + `window_id` posts `WM_RBUTTONDOWN/UP` at the element's cached on-screen center; `x`, `y` posts to the deepest child window at that point. Exactly one addressing mode; `pid` required in both. `modifier` is accepted for macOS parity (no-op on Windows).

<Callout type="info">
  Right-click on Chromium **web content** via pixel coordinates is coerced to a left-click by the renderer on every synthetic path. Use `element_index` on AX/UIA-addressable targets (links, buttons, toolbar items). See [Limits](/cua-driver/reference/limits).
</Callout>

**Arguments:**

- `pid` (integer, required): Target process ID.
- `element_index` (integer, optional): Index from the last `get_window_state`. Requires `window_id`.
- `window_id` (integer, optional): HWND for the target window.
- `x` / `y` (number, optional): Window-local screenshot pixels; provide together.
- `dispatch` (string, optional): See [Dispatch modes](#dispatch-modes).
- `from_zoom` (boolean, optional): Coordinates are in the last `zoom` image for this pid.
- `modifier` (array of string, optional): Accepted for parity; no-op on Windows.

```json
{"pid":844,"window_id":10725,"element_index":14}
```

### drag

Press-drag-release gesture from (`from_x`, `from_y`) to (`to_x`, `to_y`) in window-local screenshot pixels. Use it for marquee selection, drag-and-drop, resizing via a handle, or scrubbing a slider. `duration_ms` (default 500) is the wall-clock budget; `steps` (default 20) interpolates intermediate `WM_MOUSEMOVE` events along the path. No focus steal. After a `zoom`, pass `from_zoom=true`.

**Arguments:**

- `pid` (integer, required): Target process ID.
- `from_x` (number, required): Drag-start X in window-local screenshot pixels.
- `from_y` (number, required): Drag-start Y.
- `to_x` (number, required): Drag-end X.
- `to_y` (number, required): Drag-end Y.
- `button` (string, optional): `left` (default), `right`, `middle`.
- `duration_ms` (integer, optional): Wall-clock duration of the path. Default 500.
- `steps` (integer, optional): Intermediate move events. Default 20.
- `dispatch` (string, optional): See [Dispatch modes](#dispatch-modes).
- `from_zoom` (boolean, optional): Coordinates are in the last `zoom` image for this pid.
- `modifier` (array of string, optional): Modifier keys held across the gesture.
- `window_id` (integer, optional): HWND the coordinates were measured against. Driver picks the frontmost window of `pid` when omitted.

```json
{"pid":844,"from_x":120,"from_y":240,"to_x":420,"to_y":520}
```

### scroll

Scroll the target pid's focused region. On Windows the transport is `WM_VSCROLL` / `WM_HSCROLL` posted to the window — the same events the OS sends for scrollbar / trackpad scrolling, so any window that handles scrollbars responds. (macOS synthesizes PageDown / arrow keystrokes; both reach backgrounded windows.) `by: "page"` maps to `SB_PAGE*` × amount; `by: "line"` to `SB_LINE*` × amount.

**Arguments:**

- `pid` (integer, required): Target process ID.
- `direction` (string, required): `up`, `down`, `left`, or `right`.
- `amount` (integer, optional): Number of scroll ticks (1–50). Default 3.
- `by` (string, optional): `line` (default) or `page`.
- `dispatch` (string, optional): See [Dispatch modes](#dispatch-modes).
- `element_index` (integer, optional): Accepted for cross-platform parity; currently no-op on Windows (UIA SetFocus not wired up yet).
- `window_id` (integer, optional): HWND of the target window. Required when `element_index` is used; otherwise auto-resolves the pid's first visible window.

```json
{"pid":844,"direction":"down"}
```

### move_cursor

Move the **agent cursor overlay** to (x, y). Does NOT move the real mouse cursor — this is purely a visual cue for demos and recordings.

**Arguments:**

- `x` (number, required): X in screen points.
- `y` (number, required): Y in screen points.
- `cursor_id` (string, optional): Multi-cursor instance id. Default `default`.

```json
{"x":100,"y":200}
```

### zoom

Zoom into a rectangular region of a window screenshot at full (native) resolution. Use it when `get_window_state` returned a resized image and you need to read small text or verify UI details. Coordinates `x1, y1, x2, y2` are in the same pixel space as the `get_window_state` screenshot; the maximum region width is 500 px in scaled-image coordinates. A 20% padding is added on every side. After zooming, pass `from_zoom=true` to `click` / `type_text` to translate coordinates back to full-window space.

**Arguments:**

- `pid` (integer, required): Target process ID (scopes the zoom-translation registry).
- `window_id` (integer, required): HWND of the target window.
- `x1` (number, required): Left edge (resized-image pixels).
- `y1` (number, required): Top edge.
- `x2` (number, required): Right edge.
- `y2` (number, required): Bottom edge.

```json
{"pid":844,"window_id":10725,"x1":120,"y1":240,"x2":520,"y2":480}
```

## Keyboard and text input

### press_key

Press and release a single key, delivered to the target pid's top-level window via `PostMessage(WM_KEYDOWN/WM_KEYUP)`. The target does NOT need to be frontmost — no focus steal. Key vocabulary: return, tab, escape, up/down/left/right, space, delete, home, end, pageup, pagedown, f1-f12, plus any letter or digit. For true combinations (ctrl+c) use [`hotkey`](#hotkey).

**Arguments:**

- `pid` (integer, required): Target process ID.
- `key` (string, required): Key name.
- `modifiers` (array of string, optional): Modifier names held while the key is pressed (ctrl/shift/alt/win on Windows; cmd/shift/option/ctrl/fn on macOS).
- `dispatch` (string, optional): See [Dispatch modes](#dispatch-modes).
- `element_index` (integer, optional): Accepted for parity; currently no-op on Windows.
- `window_id` (integer, optional): HWND for the target window. Auto-resolves the pid's first visible window when omitted.

```json
{"pid":844,"key":"return"}
```

### hotkey

Press a combination of keys simultaneously — e.g. `["ctrl", "c"]` for Copy. Dispatch is target-aware on Windows:

- **Modern XAML / WinUI / UWP targets** route through UI Automation: the driver finds a descendant whose `AcceleratorKey` matches the combo and invokes it. No focus steal.
- **Legacy Win32 targets with modifiers** (Ctrl+S, etc.) route through `SendInput` with a brief `SetForegroundWindow` swap, because `PostMessage(WM_KEYDOWN, VK_CONTROL)` doesn't update the OS-wide modifier state that `TranslateAccelerator` reads. Requires the daemon to have UIAccess integrity (the MCP proxy auto-prefers the `cua-driver-uia.exe` worker when both pipes are up).
- **Legacy Win32 targets without modifiers** (plain `enter`, `f5`) route through `PostMessage` — no focus steal.

Recognized modifiers: ctrl/control, shift, alt, win/windows (cmd/command, shift, option/alt, ctrl/control, fn on macOS). Order: modifiers first, one non-modifier last.

**Arguments:**

- `pid` (integer, required): Target process ID.
- `keys` (array of string, required): Modifier(s) and one non-modifier key, e.g. `["ctrl", "c"]`.
- `dispatch` (string, optional): See [Dispatch modes](#dispatch-modes).
- `window_id` (integer, optional): Explicit HWND when the pid owns multiple windows.

```json
{"pid":844,"keys":["ctrl","c"]}
```

### type_text

Insert text into the target pid via character-by-character `PostMessage(WM_CHAR)` to the focused window. No focus steal. Special keys (Return, Escape, arrows, Tab) go through `press_key` / `hotkey` — they are not text.

When the target is a XAML / WinUI3 / UWP host (modern Notepad, Calculator, Photos, Settings, …), the tool requires `element_index` + `window_id` and routes through UI Automation's `ValuePattern.SetValue` — the same backend as `set_value` — because `WM_CHAR` doesn't reach those hosts. Calling `type_text(pid, text)` on a XAML host without `element_index` returns an actionable error pointing you at `get_window_state` first. Legacy Win32 apps keep the PostMessage path.

**Arguments:**

- `pid` (integer, required): Target process ID.
- `text` (string, required): Text to insert at the focused element's cursor.
- `delay_ms` (integer, optional): Milliseconds between characters on the PostMessage path (0–200). Default 30. Ignored on the UIA path (SetValue is atomic).
- `dispatch` (string, optional): See [Dispatch modes](#dispatch-modes).
- `element_index` (integer, optional): Required for XAML / UWP hosts (UIA path); accepted-but-optional elsewhere.
- `window_id` (integer, optional): HWND of the target window. Required when `element_index` is used.

```json
{"pid":844,"text":"hello"}
```

### set_value

Set a value on a UIA element via the `ValuePattern` interface (`AXValue` on macOS). Two modes: **standard input** (text fields, sliders, combo-box edit) writes directly through `IUIAutomationValuePattern::SetValue`; **ComboBox / select dropdown** picks the option whose text matches `value`. For free-form text that the target only accepts via keystrokes, prefer `type_text` — ValuePattern writes are ignored by some web inputs.

**Arguments:**

- `pid` (integer, required): Target process ID.
- `window_id` (integer, required): HWND of the window. Scopes the `element_index` cache per `(pid, window_id)`.
- `element_index` (integer, required): Index from the last `get_window_state`.
- `value` (string, required): New value. UIA coerces it to the element's native type.

```json
{"pid":844,"window_id":10725,"element_index":14,"value":"42"}
```

## App and window lifecycle

### launch_app

Launch an app hidden — the driver never brings the target to the foreground (`SW_SHOWNOACTIVATE` on Windows), so it does not steal focus from whatever is currently frontmost. Returns the launched pid, name, active flag, and a `windows` array (same shape as `list_windows`), so callers can skip a `list_windows` round-trip before `get_window_state`. When the launch settles but no window has materialized yet (rare), `windows` comes back empty — call `list_windows(pid)` a moment later.

Targeting:

- **Windows:** provide `bundle_id` / `name` / `aumid` (packaged apps) or `path` (full `.exe` path). An `aumid` (or `bundle_id` containing `!`) activates the packaged app via `IApplicationActivationManager` and returns the real packaged-process pid — required on Win11 for built-in apps like Notepad, Calculator, and Paint, whose `System32` `.exe` is a stub. `path` wins over `name`.
- **macOS:** provide `bundle_id` (preferred, e.g. `com.apple.calculator`) or `name`.
- `urls` opens each URL without activating the app.

**Arguments:**

- `name` (string, optional): App display name.
- `bundle_id` (string, optional): macOS bundle id, or a Windows AUMID (`{PackageFamilyName}!{ApplicationId}`).
- `aumid` (string, optional): Explicit Windows AUMID. Takes precedence over `bundle_id` / `name`.
- `path` (string, optional): Full path to an executable (Windows). Wins over name/bundle_id/aumid.
- `launch_path` (string, optional): Round-trip the `launch_path` from `list_apps`. Highest precedence.
- `urls` (array of string, optional): URLs to open without activation.
- `additional_arguments` (array of string, optional): Extra command-line / activation arguments.
- `start_minimized` (boolean, optional): Launch minimized to the taskbar so the user's frontmost window stays visually on top. UIA / background dispatch still work; only screenshots and `dispatch:"foreground"` need it restored.

The macOS-specific `electron_debugging_port`, `webkit_inspector_port`, and `creates_new_application_instance` fields are accepted on Windows for parity but currently no-op.

```json
{"name":"Calculator"}
```

### bring_to_front

Activate `pid`'s window (or `window_id` if specified) — bring it to the OS foreground. **This deliberately breaks the no-foreground contract.** Use it only when an agent is about to drive a sequence of operations against a target whose input stack drops backgrounded events (Chromium DOM content, GTK widgets) and wants to pay the foreground cost once instead of per call: call `bring_to_front`, then issue subsequent input calls with `dispatch:"foreground"` (the `SendInput` swap is then a no-op since the target is already foreground). Returns `{previous_fg_hwnd, now_fg_hwnd}` so callers can restore. Windows-only; macOS / Linux return a pointer to the platform-native alternative.

**Arguments:**

- `pid` (integer, required): Target process ID.
- `window_id` (integer, optional): HWND. Defaults to the pid's first visible top-level window.

```json
{"pid":844}
```

### kill_app

Force-terminate a process by pid. Use it when the polite close path (Alt+F4 / clicking the X) fails to make the process exit — typical for UWP / WinUI3 apps (Calculator, Photos, modern Notepad) that route `WM_CLOSE` into a suspended-but-resident state. Equivalent to `taskkill /F /PID <pid>`. Unsaved state is lost; prefer the click-the-X path first.

**Arguments:**

- `pid` (integer, required): PID of the process to terminate.

```json
{"pid":844}
```

## Browser pages

### page

Interact with the browser page loaded in a running app. Supports Chrome, Brave, Edge, and Safari (AppleScript on macOS), Electron apps (CDP), Chromium/Firefox on Windows (UIA for reads; CDP for `execute_javascript` when launched with `--remote-debugging-port`), and WKWebView / Tauri / AT-SPI fallbacks. The browser does not need to be frontmost for read actions.

Actions:

- `execute_javascript` — run JS and return the result.
- `get_text` — extract visible page text.
- `query_dom` — find elements matching a CSS selector.
- `click_element` — click a CSS-selected element and animate the agent cursor to its center first (visible feedback). Prefer over `execute_javascript('el.click()')` when you want the cursor cue.
- `enable_javascript_apple_events` — macOS-only; patches the browser's Preferences to allow JS from Apple Events (requires user confirmation and a browser restart).

**Arguments:**

- `action` (string, required): One of the actions above.
- `pid` (integer, optional): Target process ID.
- `window_id` (integer, optional): Target window from `list_windows`.
- `javascript` (string, optional): JS for `execute_javascript`.
- `css_selector` (string, optional): Selector for `query_dom`.
- `selector` (string, optional): Selector for `click_element`.
- `attributes` (array of string, optional): Element attributes to include in `query_dom` results.
- `bundle_id` (string, optional): Browser bundle id (required for `enable_javascript_apple_events`, macOS only).
- `user_has_confirmed_enabling` (boolean, optional): Must be true to proceed with `enable_javascript_apple_events`.

```json
{"action":"get_text","pid":844,"window_id":10725}
```

## Agent cursor

### set_agent_cursor_enabled

Toggle the visual agent-cursor overlay. When enabled, pointer actions animate a floating arrow to the target's on-screen position before firing — purely visual, the dispatch itself is unchanged. Default: enabled. Disable with `{"enabled": false}` for headless / CI runs.

**Arguments:**

- `enabled` (boolean, required): True to show the overlay; false to hide.
- `cursor_id` (string, optional): Multi-cursor instance id. Default `default`.

```json
{"enabled":false}
```

### set_agent_cursor_motion

Configure a cursor instance's appearance and motion curve. All fields optional; only the knobs you pass change.

Appearance: `cursor_icon` (built-in `arrow`/`crosshair`/`hand`/`dot` or a PNG/SVG path), `cursor_color` (hex or CSS name), `cursor_label`, `cursor_size` (default 16), `cursor_opacity` (0.0–1.0, default 0.85).

Motion (cubic Bezier + settle spring): `arc_size` (default 0.25), `arc_flow` (asymmetry bias [-1,1], default 0), `start_handle` / `end_handle` (default 0.3), `spring` (settle damping [0.3,1.0], default 0.72), `glide_duration_ms` (flight per move [50,5000], default 160), `dwell_after_click_ms` (default 80), `idle_hide_ms` (auto-hide delay [0,60000], 0 = never, default 20000).

**Arguments:**

- `cursor_id` (string, optional): Instance name. Default `default`.
- `cursor_icon` / `cursor_color` / `cursor_label` (string, optional): Appearance.
- `cursor_size` / `cursor_opacity` (number, optional): Appearance.
- `arc_size` / `arc_flow` / `start_handle` / `end_handle` / `spring` (number, optional): Motion-curve knobs.
- `glide_duration_ms` / `dwell_after_click_ms` / `idle_hide_ms` (number, optional): Timing knobs.

### set_agent_cursor_style

Update the cursor overlay's visual style. All fields optional.

**Arguments:**

- `gradient_colors` (array of string, optional): CSS hex stops tip→tail (e.g. `["#FF0000","#0000FF"]`). `[]` reverts to default.
- `bloom_color` (string, optional): Hex color for the halo/bloom. `""` reverts to default.
- `image_path` (string, optional): Path to a PNG/JPEG/SVG/ICO icon to use instead of the gradient arrow. `""` reverts to the arrow.
- `cursor_id` (string, optional): Instance. Default `default`.

### get_agent_cursor_state

Report the current agent-cursor configuration: enabled flag, motion knobs, glide duration, post-click dwell, and idle-hide delay. Durations come back in milliseconds. Read-only.

**Arguments:** none.

## Recording and replay

### start_recording

Start trajectory recording. Every subsequent action-tool call (click, right_click, scroll, type_text, press_key, hotkey, set_value) writes a turn folder under `output_dir`: `app_state.json` (post-action snapshot), `screenshot.png`, `action.json` (tool name, arguments, result, pid, click point, timestamp), and `click.png` (click-family actions, with a dot at the click point). Turn folders are `turn-00001/`, `turn-00002/`, …; numbering restarts each time recording starts.

**Video is on by default** — the main display is captured to `<output_dir>/recording.mp4` (H.264 / 30 fps). Pass `record_video: false` to opt out. macOS uses native ScreenCaptureKit (no extra TCC prompt, macOS 15.0+); Windows and Linux use an ffmpeg subprocess (`gdigrab` / `x11grab` + libx264, ffmpeg on PATH). When ffmpeg is missing, per-turn capture still runs and the session's `last_error` carries the diagnostic. Call `stop_recording` to finalize the mp4.

**Arguments:**

- `output_dir` (string, required): Absolute or `~`-rooted directory for turn folders and the video.
- `record_video` (boolean, optional): Capture the main display to `recording.mp4`. Default true.

```json
{"output_dir":"~/cua-trajectories/demo1"}
```

### stop_recording

Stop trajectory recording. Disables further per-turn capture and, when video was enabled, gracefully terminates ffmpeg so the mp4 is playable. A no-op on an already-stopped session. The response carries `last_video_path` when video was on.

**Arguments:** none.

### get_recording_state

Report the recorder state: whether recording is enabled, the output directory, and the 1-based counter for the next turn folder. Read-only.

**Arguments:** none.

### replay_trajectory

Replay a recorded trajectory by re-invoking every turn's tool call in lexical order. `dir` must point at a directory written by `start_recording`; each `turn-NNNNN/action.json` is replayed through the same dispatch path a live call uses.

Caveats: element-indexed actions fail (indices are per-snapshot and don't survive sessions) — pixel clicks and keyboard tools replay cleanly. Read-only tools aren't recorded, so the element cache isn't repopulated. If recording is enabled during replay, the replay is itself recorded — deliberate, for diffing a trajectory against a new build.

**Arguments:**

- `dir` (string, required): Trajectory directory from `start_recording`. Absolute or `~`-rooted.
- `delay_ms` (integer, optional): Milliseconds between turns. Default 500.
- `stop_on_error` (boolean, optional): Stop on the first error. Default true.

```json
{"dir":"~/cua-trajectories/demo1"}
```

## Config and updates

### get_config

Report the current persistent driver config. Read-only; returns defaults when unset. On Windows the shape is `{schema_version, version, platform, capture_mode, max_image_dimension}`.

**Arguments:** none.

### set_config

Write a setting into the persistent config; values take effect immediately. Two input shapes: the dotted-path form `{"key": "capture_mode", "value": "som"}` (preferred), or the bulk per-field form `{"capture_mode": "som", "max_image_dimension": 0}`.

Known keys: `capture_mode` (`vision` | `ax` | `som`), `max_image_dimension` (integer), `experimental_pip` (boolean; applies on next daemon restart — Windows backend stubbed today), `experimental_pip_geometry` (string `WxH` or `WxH+X+Y`). Returns the full updated config.

**Arguments:**

- `key` (string, optional): Dotted-path leaf field. Pair with `value`.
- `value` (any, optional): New value for `key`.
- `capture_mode` / `max_image_dimension` / `experimental_pip` / `experimental_pip_geometry` (optional): Per-field shape.

```json
{"key":"capture_mode","value":"som"}
```

### check_for_update

Check whether a newer `cua-driver-rs` release is available on GitHub. Returns current and latest versions, an `update_available` boolean, the install one-liner, and the release-notes URL. Read-only — never installs. Mirror of `cua-driver check-update --json`. See [Updating](/cua-driver/guide/getting-started/updating).

**Arguments:** none.

## Platform notes

- **UWP / WinUI3 / XAML targets** (Calculator, Settings, modern Notepad) self-foreground during a naive UIA Invoke. The driver wraps Invoke / Expand / Toggle in an `EnableWindow(host, FALSE)`-gated call so the pattern still delivers through the accessibility channel without the target stealing z-order. CoreWindow-hosted apps are reached via a UIA root-walk fallback. See [Autostart → UWP automation and the uiAccess worker](/cua-driver/guide/getting-started/autostart#uwp-automation-and-the-uiaccess-worker-windows).
- **Chromium browsers** launched hidden need anti-throttling flags so the occluded renderer keeps rendering and exposing its DOM; the driver auto-injects them. For coord clicks into DOM content, `SendInput` (`dispatch:"foreground"`) or a `--remote-debugging-port` + the `page` tool (CDP) is required — `PostMessage` doesn't reach Chromium's input thread.
- **macOS Swift driver** uses Accessibility, `CGEvent` / SkyLight, and ScreenCaptureKit, and exposes a standalone `screenshot` tool plus Space-aware fields (`on_current_space`, `space_ids`). See the [process model](/cua-driver/guide/getting-started/process-model) and [Limits](/cua-driver/reference/limits).
